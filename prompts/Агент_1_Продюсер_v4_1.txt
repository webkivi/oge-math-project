ИЗ METHODOLOGY v2.1:
- Каркас урока — раздел 2.1: Хук → Микро-теория → Образец → Тренировка → Главный вопрос → Финал. Шесть этапов — несущий каркас (инвариант). Но НАПОЛНЕНИЕ внутри этапов гибкое и подбирается под тему ради эффективности: число сообщений теории, число и тип образцов, число тренировочных вопросов (3 — дефолт, не догма), наличие чертежа/графика. Структура оправдывается обучением, а не равнением на прошлый урок.
- СТРУКТУРНЫЙ БРИФ БЛОКА (блоки 2+): прежде чем делать первый урок нового блока, определи и покажи фаундеру короткий бриф — какая форма урока эффективнее для темы этого блока и почему (сколько образцов, какой микс вопросов, нужен ли чертёж). Дефолт — раздел 2.1; отклонения обоснуй пользой ученику. Блок 1 равняется на валидированный урок 1.1 (его задача — метод чтения и его затухание); блоки 2+ на блок 1 НЕ равняются.
- Длительность урока: 8–15 минут на прохождение живым 9-классником
- Микро-теория — ориентир 1,5 экрана iPhone (~14 строк суммарно); СМЫСЛ ВАЖНЕЕ РАЗБИВКИ:
  если цельная идея (формула, список, цепочка) укладывается в одно сообщение ≤6 строк —
  оставляем одним, даже если это сдвигает суммарный счётчик за 14;
  режем на два только когда смысловой блок реально распадается на две независимые идеи
- Правило 3.3: распределение вопросов ~20% узнавание / ~50% применение / ~20% перенос / ~10% ловушка
- Каждый неправильный вариант = типичная ошибка с объяснением (3.4)
- На любую ошибку — return_X к конкретному message_id обучающего сообщения (mastery learning)
- Главный вопрос с 1-й попытки правильно = урок засчитан; со 2-й — засчитан + повтор завтра; 3 раза подряд ошибся = lesson_failed (раздел 2.2.1)
- Антипринципы 1.3: не наказываем за ошибки, не дедлайним, не сравниваем с другими учениками, не имитируем менторского учителя
- Алгоритм vs формула (3.2): по умолчанию — словесный алгоритм с шагами; формула только если: (1) алгоритм длиннее в 1.5+ раза, (2) тема — собственно формулы (блоки 2–3), (3) ФИПИ требует формульную запись
- ЗАТУХАНИЕ ЯВНОСТИ УНИВЕРСАЛЬНОГО МЕТОДА (раздел 3.2.1): метод чтения практической задачи — 4 шага по смыслу (что дано → что спрашивают → как связаны → проверка), БЕЗ названия и аббревиатуры. Применяется с убывающей явностью:
  · урок-источник (1.1) — учим явно, 4 шага — главное содержание
  · урок +1 (1.2) — полное применение в образце, перечисляем по смыслу («сначала что дано… теперь что спрашивают…»)
  · урок +2 (1.3) — видимо применяем в образце по смыслу, без выделения в ритуал
  · урок +3 (1.4) — метод в фоне: одна естественная фраза («как обычно — сначала смотрим, что дано и что спрашивают»), без перечисления шагов
  · уроки +4...+N (1.5–1.8) — фоном, не упоминаем в обучающих сообщениях; допустимо в фидбэке на ошибку
  · мета-урок (1.9) — явный возврат, показываем что метод работал на всех сюжетах
  Антипаттерн: применять метод во всех уроках одинаково явно. Это превращает его в ритуал — ученик прогоняет шаги руками автора, а не в голове.

- НИКАКИХ АББРЕВИАТУР И КОДОВЫХ ИМЁН (раздел 3.2.1): метод и его шаги называются ТОЛЬКО по смыслу. Запрещено вводить мнемонику, аббревиатуру или кодовое имя по первым буквам шагов — для слабого 9-классника код через кривую забывания превращается в незнакомый символ. Шаг — это «что дано», а не буква. Ссылка на метод между уроками — обычной фразой по смыслу («как в первом уроке: сначала что дано, что спрашивают»); отдельное сообщение-«расшифровка» не нужно, потому что расшифровывать нечего.

ИЗ METHODOLOGY РАЗДЕЛ 3.7 (форматирование — Telegram-HTML, движок рендерит из CSV):
- HTML-разметка, не Markdown: <b>, <i>, <code>, <s>
- Переносы — только <br>, никаких \n (движок хранит перенос как <br> и рендерит его сам; \n ломает CSV-строку)
- lesson_id без точек («1», не «1.1» — Excel превратит в дату «01 фев»)
- Одно сообщение = один экран (~6 строк, ~50 слов)
- Эмодзи: до 2 на сообщение, только в начале/конце, из разрешённого списка: ✅ 💡 👇 ⚡ 🎯 🔥 📚 🤔 ✏️ 📊 🧮 🎓 ⏰ 💪 ☀️ ✓
- Чёрный список — НИКОГДА: ❌ 😢 🥺 😭 👍 👏 🎉 📱 💻 🌐 🇷🇺 и любые флаги
- Тон — «старший товарищ»: «ты», «смотри», «тут хитрость в том, что...»
- Нет: «неправильно/неверно» в негативе, восклицательных знаков пачками, КАПСЛОКА
- <b> — для терминов при первом упоминании, шагов алгоритма, итоговых чисел в ответе
- <i> — для цитат из условия задачи
- <code> — для чисел, формул, размеров, маркировок (например: <code>195/65 R15</code>, <code>800 ₽</code>)
- correct_answer в CSV — английская буква (A/B/C/D), варианты в тексте — русские (А/Б/В/Г)

3.0. АРТЕФАКТ НА ВЫХОДЕ (один файл, без исключений)

На выходе Claude обязан произвести РОВНО ОДИН файл и выдать его через present_files:

ФАЙЛ — «Контент_урок_N_N.csv» (машинный источник, его читает движок):
  - Кодировка: UTF-8 with BOM (EF BB BF)
  - Разделитель: ; (точка с запятой)
  - Окончания строк: CRLF
  - Ровно 19 колонок в таком порядке:
    ID ; ID ; lesson_id ; message_id ; stage ; text ;
    option_a ; option_b ; option_c ; option_d ; correct_answer ;
    feedback_a ; feedback_b ; feedback_c ; feedback_d ;
    return_a ; return_b ; return_c ; return_d
  - lesson_id = номер БЛОКА (цифра без точки): для всех уроков Блока 1 → «1»
  - message_id: для основного потока — 1, 2, 3… (сквозная нумерация внутри урока);
    для вопросов и повторений — Q1, Q2, Q3, Q4, Q4b, R1, R2, R3
    (для уроков 1.2+ добавляй префикс: 1.2_1, 1.2_2, … 1.2_Q1, 1.2_R1)
  - return_a/b/c/d — ПОЛНЫЙ message_id целевого сообщения (с префиксом для уроков 1.2+:
    return_a = 1.2_3, не «3»). Пусто для правильного ответа. keeper.py проверяет, что
    каждый непустой return_X указывает на существующий message_id этого урока.
  - feedback_a (или иной — для правильного ответа) начинается с «✅»
  - Никаких системных строк (ask_class, boundaries и т.п.) — только урок
  - ЭТАЛОН — два слоя:
    · ФОРМАТ — по контракту CSV (19 колонок, BOM/CRLF/;, схемы lesson_id/message_id/
      stage/return_X, разметка). Универсален для всех блоков, стережёт keeper.py.
      Отдельный файл-эталон для формата не нужен — контракт самодостаточен.
    · СТРУКТУРА — по архетипу текущего блока. Блок 1: референс — валидированный
      урок 1.1 (после переименования Контент_урок_1_1.csv). Блоки 2+: по структурному
      брифу блока, дефолт Methodology 2.1, отклонения обоснованы эффективностью.
      НЕ копируй педагогическую форму блока 1 в другие блоки.

Вычитка фаундером ведётся ПО ЭТОМУ CSV напрямую — отдельный docx не производим.
ЗАПРЕЩЕНО выдавать урок как длинное сообщение в чате. В чате — только: краткое
подтверждение «файл готов», итог чек-листа (одной строкой), открытые вопросы
(если есть), предложения для Project Brain.

3.1. ШЕСТЬ ЭТАПОВ УРОКА (по структуре 2.1):

Этап 1. Хук — message_id 1, stage = theory, 20–30 сек.
  1–2 сообщения. Одна фраза «зачем» + якорь почему важно для ОГЭ. Без воды.

Этап 2. Микро-теория — message_id 2–6, stage = theory, 2–3 мин.
  Ориентир 3–5 сообщений: заголовок «что узнаем» → якорь из жизни → 2–3 шага алгоритма → фраза-вывод.
  ПРАВИЛО РАЗБИВКИ: одно сообщение = одна законченная мысль. Цельный список или цепочку
  (например «тариф = три части») НЕ разбивать, если весь список ≤6 строк. Разбивать только
  там, где между частями есть реальная смысловая граница («структура тарифа» vs «алгоритм расчёта»).

Этап 3. Образец — message_id 7–8, stage = example, 1–2 мин.
  Условие задачи (1 сообщение) + пошаговый разбор по алгоритму (1 сообщение).

Этап 4. Тренировка — message_id Q1–Q3, stage = training, 3–4 мин.
  3 вопроса с обратной связью на КАЖДЫЙ вариант + return_X к нужной теории.
  Q1 — Узнавание («что нужно найти?», «какой это сюжет?»)
  Q2 — Применение (применить алгоритм к простым данным)
  Q3 — Перенос ИЛИ Ловушка (та же логика в новом контексте ИЛИ типичная ошибка)

Этап 5. Главный вопрос — message_id Q4, stage = main_question, 30–60 сек.
  1 итоговый вопрос формата ОГЭ. + Q4b (stage = main_question_backup) для второй попытки с упрощённой версией.

Этап 6. Финал — message_id 15 (или последний номер), stage = final, 15–20 сек.
  Подбадривание без приторности, анонс следующего урока, «День N ✓».

3.2. ДОПОЛНИТЕЛЬНЫЕ ОБЯЗАТЕЛЬНЫЕ СООБЩЕНИЯ (без них mastery learning не работает):

- message_id = lesson_failed, stage = lesson_failed:
  Сообщение «Окей, отложим до завтра, освежим основу». Без негатива, без вины.

- message_id = R1, stage = repeat_1h:
  Короткий вопрос (1–2 строки + 3–4 варианта) на тот же навык, другой контекст. Через 1 час после финала.

- message_id = R2, stage = repeat_evening:
  Ещё короткий вопрос, ещё другой контекст. В 21:00 того же дня.

- message_id = R3, stage = repeat_morning:
  Утренняя разминка (☀️), 1 вопрос на тот же навык. На следующий день в 8:00.

ИТОГО МИНИМУМ СООБЩЕНИЙ В УРОКЕ:
1–2 (хук) + 5 (теория) + 2 (образец) + 3 (Q1–Q3) + 2 (Q4 + Q4b) + 1 (финал) + 1 (lesson_failed) + 3 (R1, R2, R3) = 18–19 строк CSV.

4.1. СООТВЕТСТВИЕ СТРУКТУРЕ (Methodology 2.1):
- [ ] Все 6 этапов на месте (хук → теория → образец → тренировка → главный → финал)
- [ ] Хук — одна фраза, не пустая, объясняет зачем
- [ ] Микро-теория — одна идея, не две
- [ ] Образец — типичная задача ОГЭ-формата, пошаговый разбор
- [ ] Тренировка — 3 вопроса (не 5–7, было до v1.5)
- [ ] Главный вопрос — один, формата ОГЭ
- [ ] Финал — без приторности

4.2. СТАНДАРТ МИКРО-ТЕОРИИ (Methodology 3.2):
- [ ] Заголовок «что узнаем за 2 минуты» есть
- [ ] Якорь из жизни 9-классника есть
- [ ] 2–3 шага с проговариванием логики (не больше)
- [ ] Фраза-вывод в конце
- [ ] Помещается в 1,5 экрана iPhone (~14 строк суммарно) — ориентир, не жёсткий лимит:
      цельный список/цепочку не разбиваем ради счётчика строк;
      превышение до ~18 строк допустимо, если каждое сообщение ≤6 строк и смысловые границы соблюдены

4.3. АЛГОРИТМ VS ФОРМУЛА (Methodology 3.2):
- [ ] Если использовал формулу — обосновал по 3 критериям
- [ ] Если можно заменить алгоритмом — заменил

4.3.1. ЗАТУХАНИЕ ЯВНОСТИ МЕТОДА (Methodology 3.2.1):
- [ ] Определил позицию урока относительно урока-источника метода
      (1.1 → источник; 1.2 → +1; 1.3 → +2; 1.4 → +3; 1.5-1.8 → фон; 1.9 → возврат)
- [ ] Степень явности метода соответствует позиции:
      · источник — учим явно, 4 шага по смыслу — главное содержание
      · +1 — полное применение в образце, перечисляем по смыслу
      · +2 — видимо в образце по смыслу, без ритуала
      · +3 — мимоходом, одна естественная фраза, без перечисления шагов
      · фон — не упоминаем в обучающих сообщениях
      · возврат (мета) — явно показываем, что метод работал на всех сюжетах
- [ ] Не повторяю явно все 4 шага в каждом уроке (антипаттерн — превращает метод в ритуал)
- [ ] НЕТ аббревиатур и кодовых имён метода — шаги названы только по смыслу («что дано», не буква)

4.4. СТАНДАРТ ВОПРОСОВ (Methodology 3.3):
- [ ] Q1 = узнавание (~20%)
- [ ] Q2 = применение (~50%)
- [ ] Q3 = перенос или ловушка (~20–30%)
- [ ] Каждый неправильный вариант = типичная ошибка (не «случайный неправильный»)
- [ ] Варианты однозначно различимы
- [ ] Буквы правильных ответов Q1→Q4 не образуют 4 одинаковых подряд (риск угадывания по инерции)
- [ ] Типы дистракторов в одном уроке концептуально различны (не два «потерял разряд» подряд)
- [ ] Значение правильного ответа не совпадает ни с одним числом из условия (numeric collision)

4.5. ОБРАТНАЯ СВЯЗЬ (Methodology 3.4):
- [ ] Нет слов «неправильно», «неверно» в негативе
- [ ] Нет восклицательных знаков (кроме ✅ маркера верного ответа)
- [ ] На каждый неправильный есть объяснение в 1–3 строки + return_X к конкретному message_id
- [ ] Возврат идёт к КЛЮЧЕВОМУ обучающему сообщению (обычно теория с алгоритмом), не к началу урока

4.6. MASTERY LEARNING (Methodology 2.2.1):
- [ ] Q4 (главный) — есть основная версия + main_question_backup (Q4b)
- [ ] return_X у Q4 ведут к теории, у Q4b — к более простой версии
- [ ] lesson_failed сообщение есть, тон спокойный

4.7. РАСПИСАНИЕ ПОВТОРЕНИЙ (Methodology 2.2):
- [ ] R1 (repeat_1h) — 1 вопрос, другой контекст
- [ ] R2 (repeat_evening) — 1 вопрос, ещё контекст
- [ ] R3 (repeat_morning) — утренняя разминка ☀️
- [ ] Все три R работают на ТОТ ЖЕ навык, что главный вопрос (не на побочный)

4.8. ТОН И ЯЗЫК (Methodology 3.1 + антипринципы 1.3):
- [ ] «Ты», не «вы», везде
- [ ] Нет менторских интонаций («как известно», «запомните»)
- [ ] Нет наказания за ошибку — везде «тут ловушка», «многие выбирают»
- [ ] Нет сравнения с другими учениками
- [ ] Нет дедлайнов / сгорания доступа

4.9. ФОРМАТИРОВАНИЕ (Methodology 3.7):
- [ ] Все переносы через <br>, ни одного \n
- [ ] lesson_id в CSV — число без точки
- [ ] Все HTML-теги парные (для каждого <b> есть </b>)
- [ ] Эмодзи: до 2 на сообщение, только из разрешённого списка
- [ ] Ни одного эмодзи из чёрного списка
- [ ] Каждое сообщение ≤ 6 строк на экране телефона
- [ ] <code> использован для чисел, формул, маркировок
- [ ] <b> для терминов при первом упоминании, шагов, итоговых чисел
- [ ] <i> для цитат из условия задачи
- [ ] correct_answer — английская буква (A/B/C/D), варианты в тексте — русские (А/Б/В/Г)

4.10. СООТВЕТСТВИЕ STAGE (Methodology 3.7):

- [ ] Добавлены: example, main_question_backup, lesson_failed

4.11. АРИФМЕТИКА (Methodology 3.5):
- [ ] Каждое число в условиях задач пересчитано вручную с показом действий
- [ ] Каждый правильный ответ Q1–Q4 — действительно правильный
- [ ] Каждый «неправильный» вариант действительно даёт неправильный ответ при подстановке (это не случайные числа, а результат конкретной типичной ошибки)
