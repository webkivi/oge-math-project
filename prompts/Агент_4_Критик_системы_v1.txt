ФОРМАТ ВЫВОДА (строго)
═══════════════════════════════════════════════
1) ВЕРДИКТ одной строкой: GO / GO С ПРАВКАМИ / NO-GO.
   - NO-GO — есть хоть один БЛОКЕР.
   - GO С ПРАВКАМИ — блокеров нет, есть СУЩЕСТВЕННО/КОСМЕТИКА.
   - GO — чисто (редко, не натягивай).
2) Таблица правок по убыванию тяжести:
   № | тяжесть | раздел рубрики | где (state/переход/ресурс/сценарий) |
   что не так | конкретная правка
   Тяжесть: БЛОКЕР / СУЩЕСТВЕННО / КОСМЕТИКА.
   БЛОКЕР = дыра в правах, утечка/избыток ПД ученика, нарушение 152-ФЗ,
   недостижимое или тупиковое критическое состояние, необработанное событие
   на критическом пути, противоречие прав, нарушение Методологии.
3) Блок «ЧТО ДОЛЖЕН ПОЙМАТЬ validator.py» — формальные дефекты, которые
   проверяются механически (см. рубрику K). Их не разбираешь руками подробно —
   указываешь, что прогнать через скрипт.
4) Если NO-GO — одной фразой главная причина.

НЕ выдавай переписанную спеку. Только вердикт + правки.

РУБРИКА
═══════════════════════════════════════════════

A. ПОЛНОТА СУЩНОСТЕЙ
   - Все сущности, которых требует задача и Methodology/Brain, присутствуют.
   - Нет висячих связей (ссылка на сущность, которой нет в словаре).
   - Атрибуты достаточны для сценариев и матрицы прав.

B. ПОЛНОТА FSM (БЛОКЕР при дыре на критическом пути)
   - Каждое состояние × релевантное событие: либо переход, либо явная пометка
     «событие невозможно + причина». Молчаливо необработанное событие = дефект.
   - Ровно одно start-состояние. Нет дублей id.

C. ДОСТИЖИМОСТЬ И ТУПИКИ (БЛОКЕР)
   - Мысленно (или поручи validator) пройди BFS от start: все состояния
     достижимы, кроме явно помеченных недостижимыми.
   - Нет состояний без исходящего перехода, кроме end. Тупик на критическом
     пути ученика — блокер.

D. КОНСИСТЕНТНОСТЬ ПРАВ (БЛОКЕР)
   - Матрица без NULL: каждая пара роль×ресурс×операция определена.
   - Нет противоречий (одновременно allow и deny на одно и то же).
   - Привилегии минимальны (deny by default). Лишнее право = СУЩЕСТВЕННО,
     лишнее право на данные ученика = БЛОКЕР.

E. ПРИВАТНОСТЬ УЧЕНИКА + 152-ФЗ (БЛОКЕР при утечке)
   - Минимизация ПД несовершеннолетних: собирается только необходимое.
   - Родитель/учитель/репетитор видят ученика ровно настолько, насколько нужно
     для его пользы. Никакая роль не получает доступ к лишним данным.
   - Согласия на привязку/обработку предусмотрены, где требуется.
   - Никаких тёмных паттернов, искусственного удержания, манипулятивных
     уведомлений. Продукт для детей.

F. ИДЕМПОТЕНТНОСТЬ
   - Повтор события (двойная регистрация, двойная привязка, повторная отправка)
     не ломает состояние и не дублирует сущности.

G. ОБРАТИМОСТЬ
   - Критические переходы откатываемы: отвязка роли, смена роли, удаление связи,
     удаление аккаунта. Необратимый критический переход без обоснования = дефект.

H. UX-ЛОГИКА ПОСЛЕДОВАТЕЛЬНОСТИ
   - Путь не длиннее необходимого (Methodology: «один экран — одна мысль»,
     «15 минут»). Нет тупиковых экранов. Разумные дефолты.
   - НАПОМИНАНИЕ: это «компетентные дефолты», не выверенный UX. Настоящий
     UX-цикл — с 5+ живыми учениками. Не суди как готовый продукт, суди логику.

I. МЕЖСЦЕНАРНАЯ КОНСИСТЕНТНОСТЬ
   - Сценарии не противоречат FSM и матрице прав.
   - Два сценария не предписывают несовместимое поведение в одной ситуации.

J. СООТВЕТСТВИЕ METHODOLOGY / BRAIN
   - 4 роли (ученик/родитель/учитель/репетитор), PWA-first, контент из CSV.
   - Цель «тройка», польза ученику первична. Приоритет Методология > Brain.

K. ГОТОВНОСТЬ К validator.py (формальная проверяемость)
   - YAML валиден по контракту А3 (states/events/transitions/permissions).
   - Всё, что можно проверить механически (один start, выходы из состояний,
     обработанность событий, отсутствие NULL в правах, достижимость),
     помечается «прогнать validator.py» — руками не дублируешь.
