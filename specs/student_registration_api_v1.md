CRUD над `registration_draft` из матрицы прав reg_v2 §3 (`anonymous/registration_draft` — read/write/create/delete = allow) при принятой модели «клиентский draft» исполняется **НА КЛИЕНТЕ in-memory**, без HTTP:

| `registration_draft/write` | клиент: записать `name_input` / `grade_input` / `ogeprep_answer` / `pd_consent_checked` / `policy_version_shown` в объект в памяти | `evt_name_submitted`, `evt_grade_selected`, `evt_ogeprep_*`, чекбокс согласия |

## 2. Генерация и транспорт `onboarding_session_id`

### 2.1 Решение: клиент генерит UUIDv4 (без серверного round-trip)

**Принято (транспортное решение архитектора):** `onboarding_session_id` генерируется **на клиенте как UUIDv4** в момент `registration_draft/create` (`evt_open_pwa`, вход в `name_entry`). Серверного эндпоинта «выдать ключ онбординга» НЕТ.

**Обоснование:**
1. **Согласованность с моделью.** При client-side draft сервер до submit не существует в потоке. Серверный round-trip за ключом противоречил бы принципу «нет per-step обращений к бэкенду» и инварианту «ПД/состояние онбординга не на сервере до submit».
2. **152-ФЗ минимизация.** Сервер не заводит запись/строку до согласия и submit — даже пустую «сессию онбординга». Ключ — это локальный идемпотентность-токен клиента, не серверная сущность.
3. **Достаточность UUIDv4.** 122 бита случайности → коллизия между двумя независимыми анонимами практически невозможна; этого достаточно как идемпотентность-ключа. Подмена ключа злоумышленником ничего не даёт: ключ лишь предотвращает дубль одного и того же submit (см. §5 RF-07), он не является секретом и не даёт доступа.
4. **reg_v2 §1 совместимость.** reg_v2 уже назвал `onboarding_session_id` «idempotency-key» и убрал опору на `unique(name, created_at)`. UUIDv4-на-клиенте — корректная реализация этого инварианта.

### 2.2 Транспорт ключа: заголовок `Idempotency-Key` (не поле тела)

**Принято:** ключ передаётся в HTTP-заголовке запроса E3:

```
Idempotency-Key: <uuid-v4>
```

**Обоснование выбора «заголовок vs поле тела»:**
- **Идиоматичность.** `Idempotency-Key` — отраслевой стандарт де-факто (Stripe, IETF draft `draft-ietf-httpapi-idempotency-key-header`). Он семантически отделяет «механизм защиты от дубля транзакции» от «бизнес-данных регистрации» в теле.
- **Прозрачность для middleware.** Идемпотентность-логика может жить в одном слое (зависимость/middleware на роутере E3) и читать заголовок, не разбирая тело. Тело валидируется guard'ами бизнес-слоя (§4) независимо.
- **Единый источник истины.** Чтобы исключить расхождение, ключ передаётся ТОЛЬКО в заголовке. В теле submit `onboarding_session_id` НЕ дублируется (см. §3 E3-request). Если сервер обнаружит ключ и в теле, и в заголовке с разными значениями — это 422 (защита от неоднозначности; §5).

### 2.3 Связь с Brain-механизмом записи

На успешном submit (`evt_submit_registration`) сервер пишет значение `Idempotency-Key` в **unique-колонку `User.onboarding_session_id`** (Brain 2026-06-17: «идемпотентность через unique-колонку User.onboarding_session_id», модель commit ae6e765: `String(64), nullable=True, unique=True`). Уникальный constraint на этой колонке — единственный серверный механизм дедупликации (см. §5 RF-07/RC-07). Никакой отдельной таблицы `OnboardingSession` нет (Brain: «RegistrationDraft / OnboardingSession НЕ персистентны»).

**Guard формата:** сервер валидирует, что `Idempotency-Key` присутствует и является корректным UUID (любой версии, ожидается v4). Отсутствие заголовка → 400 (отсутствует обязательный HTTP-заголовок, см. §4 G0 / §5), невалидный формат → 422 (семантически невалидное значение). Длина ключа должна укладываться в `String(64)` (UUID — 36 символов, укладывается).

## 3. Схемы request / response (JSON)

Типы: `string`, `integer`, `boolean`, `null`. `?` = поле опционально/nullable. Имена полей — snake_case (контракт с моделью SQLAlchemy commit ae6e765).

### E1. `GET /api/pd-policy`

**Request:** без тела, без аутентификации.

**Response 200:**
```json
{
  "policy_version": "string",        // актуальная версия Политики ПД (пишется в pd_consent_version при submit); серверная, ≤ 40 символов (String(40), модель ae6e765)
  "policy_url": "string",            // URL текста Политики (публичная ссылка на экране согласия)
  "available": true                  // boolean; false → чекбокс согласия неактивен (RF-04)
}
```
> `policy_version` клиент сохраняет в `RegistrationDraft.policy_version_shown` (in-memory) и при ПОВТОРНОМ входе в `consent_gate` (reg_v2 правка №6, RC-14) **перезапрашивает E1**, чтобы зафиксировать актуальную на момент показа версию. Поскольку `policy_version` — server-controlled и совпадает с `current_policy_version` (см. §4 G4), оно гарантированно укладывается в `String(40)` модели.

**Response 503** (политика недоступна, RF-04): тело `{ "error": "pd_policy_unavailable" }`; клиент держит чекбокс согласия неактивным, submit невозможен.

### E2. `GET /api/session`

**Request:** без тела; cookie `oge_session` (если есть) передаётся браузером автоматически.

**Response 200** (валидная сессия — RC-11, ученик уже зарегистрирован):
```json
{
  "authenticated": true,
  "role": "student",
  "fsm_state": "registered",          // из StudentProfile.fsm_state (v4); клиент НЕ входит в name_entry
  "current_lesson_id": "string"       // ВСЕГДА непустой: StudentProfile.current_lesson_id NOT NULL (модель ae6e765); назначен в submit детерминированно. НЕ про загрузку контента — см. §7.1
}
```

**Response 200** (нет/невалидная/истёкшая сессия — анонимный):
```json
{
  "authenticated": false
}
```
> E2 всегда отвечает 200; различие — в поле `authenticated`. Это re-auth-проба, а не защищённый ресурс (отсутствие сессии — норма для анонима, не ошибка). `current_lesson_id` присутствует ТОЛЬКО при `authenticated: true` и всегда непустой (само поле существует у зарегистрированного ученика как NOT NULL-колонка профиля); загрузка КОНТЕНТА этого урока — забота экрана урока, не E2 (§7.1).

### E3. `POST /api/registration` (submit)

**Request headers:**
```
Content-Type: application/json
Idempotency-Key: <uuid-v4>          // см. §2.2; ОБЯЗАТЕЛЕН (отсутствие → 400, §4 G0)
```

**Request body:**
```json
{
  "name": "string",                       // никнейм/отображаемое имя; НЕ ФИО (reg_v2 §1); 1..120 символов после trim (guard §4 G1/G1b, String(120) модели)
  "grade": 9,                             // integer; ожидается 9|10|11 (guard §4)
  "ogeprep_answer": "yes|no|null",        // для grade IN (10,11); для grade=9 — null by design (reg_v2 §1, правка №7)
  "pd_consent_checked": true,             // boolean; должно быть true (guard §4, 152-ФЗ)
  "policy_version_shown": "string"        // версия Политики, реально показанная ученику (из E1); пишется в pd_consent_version; ≤ 40 символов (guard §4 G4, String(40) модели)
}
```
> `onboarding_session_id` в теле НЕ передаётся — он только в заголовке `Idempotency-Key` (§2.2). `enrollment_reason` в теле НЕ передаётся — сервер ДЕРИВИТ его (§6, анти-подмена). `consent_cohort_flag` в теле НЕ передаётся — сервер выставляет сам (§6). `current_lesson_id` в теле НЕ передаётся — сервер проставляет детерминированной стартовой константой курса (§6 шаг 2, §7).

**Response 201 Created** (успех — новый аккаунт создан):
```json
{
  "user_id": "string",
  "role": "student",
  "grade": 9,
  "fsm_state": "registered",
  "enrollment_reason": "grade9_direct|grade10plus_retake",  // дериватив сервера (§6)
  "current_lesson_id": "string",          // ВСЕГДА непустой на 201: NOT NULL-колонка, назначена детерминированно в транзакции submit (§6 шаг 2, §7). Это ID урока, не его контент
  "next": "daily_start"                   // куда клиенту перейти (стыковка в v4, §7)
}
```
+ заголовок `Set-Cookie: oge_session=...` (§6).

**Response 200 OK** (идемпотентный повтор — тот же `Idempotency-Key`, аккаунт уже существует; политика §5/RF-07): тело идентично 201, но HTTP-код 200 (ресурс не создавался заново). Возвращаются уже сохранённые СЕРВЕРНЫЕ значения (включая `current_lesson_id` из существующего профиля); тело повтора НЕ применяется и НЕ сравнивается (§5.1). Cookie переустанавливается (idempotent re-auth).

**Response 4xx** — см. §5 (тело: `{ "error": "<code>", "field": "<имя поля или null>" }`).

## 4. Серверные guard'ы на submit (E3)

Бэкенд НЕ доверяет фронтенду (CLAUDE.md §6). Каждый guard дублирует клиентскую валидацию и проверяется НА СЕРВЕРЕ до старта транзакции. **Порядок проверок:**

0. **G0** (наличие/формат `Idempotency-Key`) — первым.
1. **Короткое замыкание идемпотентности (G6-lookup):** если по `Idempotency-Key` УЖЕ найден закоммиченный `User` — сервер сразу отвечает **200** существующими серверными значениями (§5.1) и НЕ прогоняет G1–G5 (включая G1b, G2-env) над телом повтора. Это критично: легитимный сетевой повтор со слегка иным телом не должен ошибочно упереться в 422.
2. Если по ключу User НЕ найден (новый ключ) — обычный порядок **G1 → G1b → G2 → G2-env → G3 → G4 → G5**, первая неуспешная определяет код ответа (§5); затем вставка под уникальным constraint (**G6** на коммите — гонка → 409, §5.1).

| # | Guard | Условие прохода | Нарушение → | Привязка |
|---|-------|-----------------|-------------|----------|
| G0 | Idempotency-Key присутствует и валиден как UUID | заголовок есть, формат UUID, длина ≤ 64 | **отсутствует заголовок → 400** `missing_idempotency_key`; **невалидный формат/длина → 422** `invalid_idempotency_key`; ключ задан и в теле, и в заголовке с расхождением → 422 `invalid_idempotency_key` | §2; RC-01, RF-07 |
| G1 | `name` непустое после `trim()` (нижняя граница) | `len(name.strip()) >= 1` | 422 `empty_name`, field=`name` | reg_v2 RC-02; FSM guard `name_entry` |
| G1b | `name` — верхняя граница и базовая нормализация | после `trim()`: `1 <= len(name) <= 120` (= `User.name` `String(120)`, модель ae6e765); нет управляющих/непечатаемых символов | 422 `invalid_name`, field=`name` | CLAUDE.md §6 «внешний ввод валидируется»; схема БД ae6e765 |
| G2 | `grade` — целое в допустимом множестве | production: `grade IN (9,10,11)`; non-prod: см. G2-env | 422 `invalid_grade`, field=`grade` | reg_v2 RC-04, RC-05; D-6 |
| G2-env | гейт grade=8 зависит от среды (см. §4.1) | в production grade=8 → reject (жёсткий гейт, User не создаётся); вне production grade=8 — dev-аффорданс v4 §1 (предупреждение, не продуктовое поведение) | production: 422 `invalid_grade` (grade=8 не создаёт аккаунт) | reg_v2 RC-04, `gate_grade8`; v4 §1 |
| G3 | `pd_consent_checked == true` | строго `true` | 422 `consent_required`, field=`pd_consent_checked` | reg_v2 RC-03; 152-ФЗ инвариант §1 |
| G4 | `policy_version_shown` непустое, длиной ≤ 40 и совпадает с действующей версией pd_policy (или допуск переподтверждения) | `1 <= len(policy_version_shown) <= 40` И `policy_version_shown == current_policy_version` (или сервер требует переподтверждения, если разошлось) | пустое/> 40 символов → 422 `invalid_policy_version`, field=`policy_version_shown`; разошлось с действующей → 409 `policy_version_mismatch` (требуется переподтверждение; экран — зона А8, см. ниже) | reg_v2 RC-08, RC-14; RF-04; `pd_consent_version` `String(40)` (модель ae6e765) |
| G5 | `ogeprep_answer` консистентен с `grade` | `grade==9 → ogeprep_answer is null`; `grade IN (10,11) → ogeprep_answer IN ('yes','no')` | 422 `inconsistent_ogeprep`, field=`ogeprep_answer` | reg_v2 §1 (ремарка №7); анти-подмена ветки |
| G6 | идемпотентность по `onboarding_session_id` | новый ключ → вставка под unique-constraint; существующий ключ перехвачен ДО G1 (шаг 1 порядка выше) | гонка на коммите → 409 `registration_conflict` (§5.1) | reg_v2 RC-07, RF-07; unique-колонка (Brain) |

> G1–G5 (включая G1b, G2-env) — валидация ВХОДА (до транзакции), выполняются ТОЛЬКО для НОВОГО ключа. G6 — конкурентный контроль на уникальном constraint (внутри/вокруг транзакции).
> **Длина `policy_version_shown` (правка №2, ревью А4 v2):** значение из тела пишется в `User.pd_consent_version` = `String(40)` (модель ae6e765). G4 проверяет верхнюю границу `len(policy_version_shown) <= 40` симметрично с G1b для `name`. На практике значение **server-controlled**: оно равно `current_policy_version`, которую сервер сам отдал на E1, и G4 требует точного равенства — поэтому корректное значение ВСЕГДА укладывается в `String(40)`. Явная граница 40 в guard'е защищает от подменённого/мусорного тела (CLAUDE.md §6 «внешний ввод валидируется»): такой ввод отклоняется 422 `invalid_policy_version` ДО попадания в БД, не вызывая обрезания/ошибки вставки.
> **G4 (409) — переподтверждение согласия, не тупик:** если на момент submit `current_policy_version` разошлась с `policy_version_shown`, сервер отвечает 409 `policy_version_mismatch` и клиент перезапрашивает E1 + переподтверждает согласие (защита целостности согласия, 152-ФЗ; согласовано с reg_v2 RC-08). **Экран переподтверждения после 409 — зона микрокопии А8** (reg_v2 RC-08/RC-14): он НЕ должен превращаться в тупик или тёмный паттерн (Methodology §1.3 «не удерживаем» искусственно; ученик всегда может уйти/отменить).

### 4.1 Env-механизм определения production (гейт grade=8)

Сервер понимает, что он «в production», через **переменную окружения** (контракт конфигурации, не секрет):

```
APP_ENV = production | staging | development     # дефолт при отсутствии = production (fail-safe)
```

Контракт:
- `APP_ENV == "production"` → G2/G2-env применяют **жёсткий гейт**: `grade IN (9,10,11)`; grade=8 → 422, User НЕ создаётся (reg_v2 `gate_grade8`, D-6 закрыта).
- `APP_ENV != "production"` (staging/development) → grade=8 — **dev-only тест-аффорданс** (v4 §1): сервер может принять grade=8 c предупреждением; это НЕ продуктовое поведение и в проде недостижимо. (Модель допускает grade IN (8,9,10,11) на уровне CHECK-constraint — это для staging; продуктовый гейт grade=8 — на guard-слое, не в БД.)
- **Fail-safe:** при отсутствии/нераспознанном значении `APP_ENV` сервер трактует среду как `production` (строжайший гейт). Это защищает от случайного открытия grade=8 из-за незаданной переменной.
- Секретов переменная не несёт (CLAUDE.md §6: секреты — только в env, но `APP_ENV` — несекретный режимный флаг; описывается в `.env.example`).

> Гейт grade=8 продублирован на бэкенде (не только UI): RC-04 — подмена grade=8 минуя UI в production отклоняется guard'ом G2-env (422), аккаунт не создаётся. В non-prod (`APP_ENV != production`) grade=8 проходит как dev-аффорданс (с предупреждением); это отражено отдельной строкой в таблице §5, чтобы non-prod-поведение не выглядело пропуском.

### 4.2 Env-механизм статуса Z-1 (источник `consent_cohort_flag`)

Признак «закрыта ли юр-зависимость Z-1» (reg_v2 §0, RF-08) сервер берёт из **конфиг-флага окружения** (несекретный режимный флаг, по аналогии с `APP_ENV`), а НЕ из имплицитной зашитой логики:

```
Z1_CONSENT_RESOLVED = true | false               # дефолт при отсутствии = false (Z-1 НЕ закрыта)
```

Контракт:
- `Z1_CONSENT_RESOLVED == false` (Z-1 ещё не закрыта; дефолт) → при `grade == 9` сервер выставляет `consent_cohort_flag = true` (помечает когорту для обратимости, reg_v2 RF-08). Для `grade != 9` — `false`.
- `Z1_CONSENT_RESOLVED == true` (юр-заключение получено, Z-1 закрыта) → `consent_cohort_flag` **перестаёт ставиться** (всегда `false`), reg_v2 RF-08 «после закрытия Z-1 `consent_cohort_flag` перестаёт ставиться».
- **Дефолт = `false`** (Z-1 не закрыта): пока юрист не подтвердил — мы продолжаем помечать когорту grade=9, чтобы негативное заключение не оказалось необратимым задним числом (reg_v2 RF-08).
- Флаг — не ПД, не секрет; описывается в `.env.example`.

> Источник статуса Z-1 теперь явный (конфиг-флаг), а не имплицитный в коде. Это транспортная фиксация механизма reg_v2 RF-08; продуктовое/юр-решение «когда Z-1 закрыта» остаётся за фаундером/юристом — флаг лишь переключается, спека не меняется.

- **422 Unprocessable Entity** — для всех **validation-guard'ов** над телом/значениями (`consent_required`, `invalid_grade`, `empty_name`/`invalid_name`, `invalid_policy_version`, `inconsistent_ogeprep`, невалидный формат `Idempotency-Key`). Это сознательный выбор поверх формулировки reg_v2 RC-03 «403/422»: тело синтаксически разобрано, но семантически невалидно → 422 идиоматичнее 403.
- **400 Bad Request** — для **отсутствующего обязательного заголовка** `Idempotency-Key` (G0): запрос некорректен на транспортном уровне (нет обязательного заголовка), а не «невалидное значение поля». ### 5.1 Политика повтора (расходящееся тело), идемпотентность vs гонка (RF-07/RC-07)

**Принято (транспортное решение):** различаем УСПЕШНО-ЗАВЕРШЁННЫЙ повтор (ключ уже закоммичен) и ОДНОВРЕМЕННУЮ гонку (никто ещё не закоммитил).

**(а) Повтор с УЖЕ СУЩЕСТВУЮЩИМ ключом → 200, тело повтора игнорируется.**
Сценарий: первый submit уже закоммитил `User` с этим `onboarding_session_id` (RF-01 повтор после сбоя сети, RC-01 дабл-клик дошёл вторым позже, RF-02 повтор после отката). Сервер находит существующего User по `onboarding_session_id` (короткое замыкание ДО G1–G5 (включая G1b, G2-env), см. порядок §4) и возвращает 200 с **уже сохранёнными СЕРВЕРНЫМИ значениями** (name, grade, enrollment_reason, current_lesson_id из существующего профиля), переустанавливает cookie. Второй User НЕ создаётся.
**Ключ привязан к ПЕРВОМУ зафиксированному результату.** Если тело повтора РАСХОДИТСЯ с первым (например пришёл `grade=8`, другое имя, иной `ogeprep_answer`) — расхождение **игнорируется**: возвращается исходный валидный аккаунт, тело повтора НЕ применяется. **Сравнение тела не требуется** — результат детерминирован первым коммитом.
- *Закрытие атаки «повтор с grade=8»:* злоумышленник, повторяющий submit с тем же ключом и `grade=8`, получит исходный валидный аккаунт (например grade=9), а не grade=8-аккаунт. Тело повтора не применяется.
- *Защита легитимного повтора:* короткое замыкание на 200 происходит ДО прогона G1–G5 (включая G1b, G2-env), поэтому легитимный сетевой повтор со слегка иным телом НЕ упрётся ошибочно в 422.
**Обоснование выбора «вернуть аккаунт, а не 409»:** UX — ученик не должен видеть ошибку из-за дабл-клика/повтора по сети; результат для него идентичен успеху; reg_v2 RF-07 явно допускает «вернуть уже созданный аккаунт ЛИБО 409» — для уже-завершённого случая выбран дружелюбный путь.

**(б) Истинная гонка → 409 `registration_conflict`.**
Сценарий: два запроса с тем же ключом стартовали почти одновременно (RC-07: две вкладки), оба не нашли существующего User на lookup-шаге, оба прошли G1–G5 (включая G1b, G2-env) и пытаются вставить, ни один ещё не закоммитил. Выигрывает один (insert проходит), проигравший ловит нарушение unique-constraint на `onboarding_session_id` и получает 409. **Обоснование:** на момент проигрыша сервер не может безопасно вернуть «готовый аккаунт» (транзакция-победитель могла ещё не зафиксироваться) — честнее отдать 409, по которому клиент сделает один повтор и попадёт в ветку (а) → 200. Итог в обоих случаях: **ровно один аккаунт на `onboarding_session_id`**.

**(в) НОВЫЙ ключ (нет существующего User) → обычный порядок.**
G1–G5 (включая G1b, G2-env) над телом, затем вставка; G6 на unique-constraint (гонка → ветка (б)).

### 5.2 RC-10 (403) — где живёт

`lesson_content` в фазе регистрации — deny для anonymous по всем операциям (reg_v2 §3). Эндпоинты чтения урока определены в v4 (`student/lesson_content/read`) и требуют валидной сессии. Анонимная попытка прочитать урок до регистрации (RC-10) отклоняется на уровне auth-зависимости v4 → 403 `forbidden`. В этом аддендуме E1–E3 такого ресурса не отдают; 403 указан для полноты привязки RC-10 к HTTP-слою и проверки А4, что deny-by-default соблюдён. 403 в этом аддендуме НЕ используется ни для одного validation-guard'а (§5 конвенция кодов).

## 6. Side-effect submit (E3, при 201)

Исполняется в **ОДНОЙ атомарной транзакции** (reg_v2 §1 инвариант; RF-02 atomicity). Либо всё, либо ничего: «висячих» User без StudentProfile/Session не возникает.

| Шаг | Действие | Источник значения | Привязка |
|-----|----------|-------------------|----------|
| 1 | Создать `User(role='student', name=name.strip(), grade=grade, created_at=now, pwa_push_token=null, pd_consent_at=now, pd_consent_version=policy_version_shown, onboarding_session_id=<Idempotency-Key>, consent_cohort_flag=<см. шаг 5>)` | тело submit + сервер | reg_v2 §1; модель ae6e765 |
| 2 | Создать `StudentProfile(user_id, fsm_state='registered', current_lesson_id=<ДЕТЕРМИНИРОВАННАЯ стартовая константа курса: lesson_id стартового урока Блока 1, из конфига курса>, course_started_at=now, last_active_at=now, enrollment_reason=<дериватив, шаг 4>)` | сервер (конфиг курса) | reg_v2 §1; v4 §1; модель `current_lesson_id` NOT NULL |
| 3 | Создать `Session(token=<256-bit random>, user_id, created_at=now, expires_at=now+30д, revoked=false)`; установить httpOnly-cookie | сервер | reg_v2 §1; v4 Session |
| 4 | Деривировать `enrollment_reason` | `grade==9 → 'grade9_direct'`; `grade IN (10,11) AND ogeprep_answer IN ('yes','no') → 'grade10plus_retake'` | reg_v2 §1, §2 (R-01/R-02/R-03) |
| 5 | Выставить `consent_cohort_flag` | `grade==9 AND Z1_CONSENT_RESOLVED == false → flag set (true)`; иначе → not set (false) | reg_v2 §1, RF-08, RC-13; §4.2 |
| 6 | Записать `User.onboarding_session_id = <Idempotency-Key>` (unique) | заголовок | Brain 2026-06-17; RF-07 |

> **Шаг 1 (правка №2, ревью А4 v2):** `pd_consent_version=policy_version_shown` пишется в колонку `String(40)` (модель ae6e765). К моменту шага 1 значение уже прошло G4: оно непустое, длиной ≤ 40 и равно `current_policy_version` (server-controlled) — поэтому укладывается в `String(40)` без обрезания. Мусорное/подменённое значение отклонено ещё guard'ом G4 (422 `invalid_policy_version`), до транзакции.
> **Шаг 2 (правка №1):** `current_lesson_id` назначается ДЕТЕРМИНИРОВАННО — это `lesson_id` стартового урока Блока 1 (урок 1.1), берётся из конфига/константы курса в той же транзакции, без внешнего вызова. Колонка NOT NULL (модель ae6e765) → в обычном потоке submit значение всегда вычисляется, INSERT не падает, аккаунт создаётся. Никакой вариативности «или null» здесь нет. Это НАЗНАЧЕНИЕ ID, не загрузка контента (§7.1).

### 6.1 Cookie сессии

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Имя | `oge_session` | стабильное имя; читается auth-зависимостью v4 |
| Значение | 256-bit random token (соответствует `Session.token`) | reg_v2 §1; v4 Session; не угадывается |
| `HttpOnly` | **да** | недоступен JS (reg_v2 §3 `session/read=false`; v4 `session_own/read=false`); защита от XSS-кражи |
| `Secure` | **да** | только по HTTPS (Brain §5.1: PWA по HTTPS; 152-ФЗ — данные детей) |
| `SameSite` | **Lax** | PWA — same-site навигация; Lax защищает от CSRF на небезопасных методах, не ломая открытие по ссылке (RC-11 re-auth по GET) |
| `Max-Age` / `Expires` | 30 дней (= `Session.expires_at - now`) | reg_v2 §1; v4 Session |
| `Path` | `/` | сессия на всё приложение |

> Cookie ставится только как side-effect успешного submit (201) или идемпотентного повтора (200) — reg_v2 §3 `session/create = true` «только как side-effect успешной регистрации». На E1/E2 cookie не выдаётся (E2 только читает существующий).

### 6.2 Анти-подмена: `enrollment_reason`, `consent_cohort_flag`, `current_lesson_id` НЕ принимаются от клиента

`enrollment_reason` сервер **деривит** из присланных `grade` + `ogeprep_answer` (шаг 4), а НЕ принимает готовым из тела. Причина: клиент мог бы прислать `grade10plus_retake` при `grade=9` или наоборот, исказив аналитику доходимости и обойдя смысл ветки. Сервер — единственный источник `enrollment_reason`. Аналогично `consent_cohort_flag` (шаг 5) выставляет только сервер по `grade` + `Z1_CONSENT_RESOLVED` (§4.2) — клиент на него не влияет (это технический флаг аудита когорты, RF-08). `current_lesson_id` (шаг 2) сервер проставляет детерминированной константой курса — клиент его не присылает и не влияет на него. `ogeprep_answer=null` для `grade=9` — by design (reg_v2 §1 ремарка №7), guard G5 это и проверяет.

## 7. Стыковка `registered → v4` (что отдаётся клиенту для перехода в первый урок)

На успешном submit (201) аккаунт в состоянии `registered` (reg_v2 терминал = v4 `registered`, normal-state). Чтобы клиент перешёл в первый урок, ответ E3 несёт:

- `fsm_state: "registered"` — каноническое состояние (v4 `StudentProfile.fsm_state`).
- `next: "daily_start"` — указание клиенту инициировать `evt_open_app` (v4 §2: `registered --evt_open_app--> daily_start`). Клиент после 201 вызывает обычный v4-поток входа в день (отдельный эндпоинт v4, вне этого аддендума), который по `evt_open_app` ведёт `daily_start → lesson_select → lesson_hook` (первый урок).
- `current_lesson_id: "<id>"` — ID первого урока курса, проставленный детерминированно в шаге 2 side-effect; **всегда непустой** (NOT NULL-колонка). Это ИД урока, а не его контент.

### 7.1 RF-06: назначение ID vs загрузка контента (правка №1)

**Два разных понятия (ключевое разнесение):**

1. **«Назначить ID первого урока»** — детерминированно, в транзакции submit, всегда успешно. `current_lesson_id = lesson_id стартового урока Блока 1` (константа/конфиг курса). Колонка `StudentProfile.current_lesson_id` — NOT NULL (модель ae6e765). В обычном потоке submit это значение НЕ может «не вычислиться» (оно не результат внешнего вызова, а константа). → На успешном `201` `current_lesson_id` ВСЕГДА присутствует и непустой. Сбоя назначения ID в happy-path submit не бывает; INSERT NOT NULL-колонки не падает.

2. **«Загрузить КОНТЕНТ первого урока на клиенте»** — отдельный, ПОЗЖЕ, на экране самого урока, уже ПОСЛЕ `registered`. Именно сюда относится **reg_v2 RF-06** («первый урок не подгрузился»): это сбой загрузки КОНТЕНТА урока на клиенте, а НЕ сбой назначения ID и НЕ часть транзакции submit. Обрабатывается на экране урока через **ConnectionStub** (v4 D-5 «нет соединения» / «урок подгрузится при следующем входе»), а не во время submit.

**Контракт API под RF-06:**
- Аккаунт уже создан (transaction committed), `current_lesson_id` уже проставлен и непустой. Регистрация НЕ откатывается (reg_v2 RF-06: «регистрация НЕ откатывается»).
- Если контент урока не подгрузился на клиенте после `registered` — это обрабатывается на экране урока (v4 D-5, ConnectionStub): показывается safe fallback «нет соединения», прогресс не теряется; контент подтянется при повторе/следующем входе (`evt_open_app`). Аккаунт и согласие сохранены.
- E3-ответ при этом неизменен: `201` с непустым `current_lesson_id` и `next: "daily_start"`. Никакого `current_lesson_id: null` в ответе submit НЕТ — это исключено решением фаундера и моделью.

> **Это КОРРЕКТИРУЕТ формулировку reg_v2 §6 RF-06 (правка №1, ревью А4 v2 — маршрут b).** reg_v2 §6 RF-06 буквально пишет «(current_lesson_id не определился)» — то есть называет сбой НАЗНАЧЕНИЯ ID, который этот аддендум показал невозможным (назначение детерминировано, всегда успешно в транзакции submit). Поэтому аддендум **замещает** скобку «(current_lesson_id не определился)» точной причиной: сбой = **загрузка КОНТЕНТА** первого урока на клиенте после `registered` (ConnectionStub, v4 D-5), а НЕ назначение ID. **Текст reg_v2 СЕЙЧАС не редактируется** этим документом; расхождение зафиксировано честно и помечено **отдельной дельтой в Project Brain через ритуал `/обновить-brain`** (пункт: «реконсилировать формулировку reg_v2 §6 RF-06 — скобку „current_lesson_id не определился“ переадресовать на загрузку контента»), на приёмку фаундером. Это НЕ «уточнение трактовки без изменения» — это честно названная коррекция формулировки reg_v2. Продуктовая развилка «когда стартует первый урок» отсутствует: курс последовательный, первый урок детерминирован (= урок 1.1). API не вводит null-fallback для `current_lesson_id` ни в одном ответе.

## 8. Edge / failure из reg_v2 на HTTP-слое

Сводка: как принятые RC/RF reg_v2 ложатся на транспорт (без переоткрытия их семантики; единственное исключение — формулировка RF-06, корректируемая правкой №1, см. ниже и §7.1).

| reg_v2 id | Что | Поведение на HTTP-слое |
|-----------|-----|------------------------|
| RC-01 | дабл-клик на «Начать» | клиент блокирует кнопку после 1-го нажатия; если 2-й запрос всё же ушёл — он несёт тот же `Idempotency-Key` → сервер: первый = 201, второй = 200 (повтор, короткое замыкание ДО G1–G5 (включая G1b, G2-env)) или 409 (гонка) → повтор → 200. Дубль User не создаётся. |
| RC-06 | закрытие браузера до submit | клиентский `RegistrationDraft` (in-memory) теряется; на сервере ничего нет (ни draft, ни User). Повторный вход = новый `RegistrationDraft` + новый `Idempotency-Key` (новый UUIDv4); submit с нуля. ПД не записаны. |
| RC-07 | гонка двух submit (две вкладки, один ключ) | unique-constraint на `onboarding_session_id`: один insert проходит (201), второй — 409 `registration_conflict` → клиент повторяет → 200. Ровно один аккаунт. (§5.1) |
| RF-01 | бэкенд недоступен в момент submit | E3 не отвечает/таймаут; клиент показывает «не удалось, попробуй ещё раз», хранит `RegistrationDraft` + тот же `Idempotency-Key` в памяти. Повтор при восстановлении сети — идемпотентен (200 если первый всё же прошёл, 201 если нет). User не «висит» (атомарность). |
| RF-02 | частичный сбой транзакции | транзакция откатывается целиком (atomicity); ни User, ни StudentProfile, ни Session не остаются. Клиент в `consent_gate` с ошибкой; повтор submit с тем же `Idempotency-Key` — идемпотентен. |
| RF-03 | офлайн при открытии PWA | service worker отдаёт кэш-оболочку онбординга (статичные экраны name/grade — они клиентские, draft in-memory); E3-submit откладывается до сети; ПД офлайн не отправляются. E1 (pd_policy) из кэша → если недоступна, чекбокс неактивен (RF-04). |
| RF-06 | **первый урок не подгрузился (КОНТЕНТ)** — формулировка reg_v2 КОРРЕКТИРУЕТСЯ (правка №1) | НЕ про submit и НЕ про назначение ID. Назначение `current_lesson_id` — детерминированно в submit, аккаунт создан (201), ID непустой. Сбой = загрузка **КОНТЕНТА** урока на клиенте ПОСЛЕ `registered`: экран урока показывает ConnectionStub (v4 D-5); регистрация НЕ откатывается; контент подтянется при повторе/следующем входе. **Аддендум замещает скобку reg_v2 §6 RF-06 «(current_lesson_id не определился)» точной причиной (загрузка контента); текст reg_v2 сейчас не меняется, дельта помечена в Project Brain через `/обновить-brain`.** (§7.1) |
| RF-07 | дубль/повтор после сбоя | см. §5.1: 200 (повтор завершённого, тело игнорируется) vs 409 (гонка) → повтор → 200. Идемпотентность по `onboarding_session_id`. |
| RF-08 | юр-заключение Z-1 негативно | когорта grade=9-до-заключения помечена `consent_cohort_flag` (ставится при `Z1_CONSENT_RESOLVED == false`, §4.2); после `Z1_CONSENT_RESOLVED == true` флаг перестаёт ставиться. Обработка когорты (согласие представителя / удаление) — reg_v2 RF-08, отдельная задача. |
