# CLAUDE.md — режим разработки

«Язык общения с фаундером — русский, всегда. Все ответы, комментарии, вопросы, отчёты — на русском. Коммит-сообщения, имена переменных и комментарии в коде — на английском (стандарт разработки).»

Версия v1 | под Methodology v2.1 / Project Brain v3.2 | 2026-06-16

Этот файл читается Claude Code в начале КАЖДОЙ сессии. Он — единственный надёжный
носитель правил разработки: Claude Code стартует с чистого контекста, личность-персона
не сохраняется, сохраняется только этот файл в репозитории. Если правило не здесь —
его в сессии нет.

## 0. Контекст проекта (одной строкой)
Адаптивная платформа микро-обучения, первый продукт — ОГЭ по математике «на тройку».
PWA-first для четырёх ролей: ученик, родитель, учитель, репетитор. Контент уроков
читается движком из CSV напрямую. Цель — доходимость, не выручка; польза ученику первична.

## 1. Источники истины и порядок чтения (ДО написания кода)
Приоритет строгий: **Methodology v2.1 > Project Brain v3.2 > спека Архитектора (А3) > этот файл.**
Перед началом любой задачи прочитай:
1. Спеку Архитектора по этой задаче (FSM, матрица прав, сценарии, edge cases).
2. Соответствующие разделы Methodology и Brain.
Если спеки на задачу нет или она неоднозначна — **СТОП**. Не импровизируй архитектуру
в коде. Иди к Архитектору (А3), получи/уточни спеку. Код без спеки на новое поведение
не пишется.

**Вики-слой (паттерн Карпати).** `specs/*_v*.md` — неизменяемые raw sources: новая
версия спеки = новый файл, старые не правятся. `specs/INDEX.md` — поддерживаемая
вики-страница: какая версия по каждой области CURRENT, какая архив, кросс-ссылки.
Открывай её перед чтением спек — быстрее, чем листать версии. Периодическая
проверка консистентности (Brain ↔ Methodology ↔ specs ↔ CLAUDE.md/AGENTS.md) —
команда `/линт-знаний`; она только докладывает находки, мастер-документы правит
фаундер (см. §9.2).

## 2. Стек и структура
- Бэкенд: Python + FastAPI. Движок контента читает CSV (19 колонок, см. контракт ниже).
- Фронтенд: React + Vite + Tailwind, service worker (PWA).
- Мессенджер-адаптеры (Telegram/MAX/VK, aiogram и др.) — этап переноса, не сейчас.
- Среда разработки: Windows + WSL2, git, Claude Code.
- Структура репозитория фиксируется первой архитектурной задачей; не меняй раскладку
  каталогов без обновления спеки.

## 3. Контракты данных (НЕ менять без обновления спеки)
Два контракта — несущие. Любое их изменение проходит через А3 (обновить спеку) → А4 →
validator.py/keeper.py, и только потом код. Молча в коде их менять запрещено.
- **CSV-контент:** UTF-8 BOM, разделитель `;`, CRLF, ровно 19 колонок в порядке
  ID;ID;lesson_id;message_id;stage;text;option_a..d;correct_answer;feedback_a..d;return_a..d.
  `lesson_id` без точек; `return_X` = полный message_id. Стережёт keeper.py.
- **FSM:** YAML по контракту А3 (states/events/transitions/permissions). Стережёт validator.py.

## 4. Конвенции кода
- Python: type hints обязательны; форматирование black + lint ruff; функции маленькие,
  имена осмысленные; никаких «магических чисел» без константы.
- JS/TS: ESLint + Prettier; React — функциональные компоненты и хуки; стили — Tailwind,
  без инлайн-стилей кроме крайних случаев.
- Имена — по-английски в коде; пользовательские тексты — не хардкодить, брать у А8
  (копирайтер) / из ресурсов.
- Каждый модуль делает одно дело. Контракты между модулями — явные.

## 5. Тесты и CI (без них фича не готова)
- Бэкенд: pytest. Фронтенд: vitest. Адаптеры: smoke-тесты.
- **Каждая фича — с тестами**, покрывающими happy path + ключевые edge cases из спеки.
- CI прогоняется на каждом коммите; красный CI = не мёржим.
- Логику FSM покрывать тестами переходов, права — тестами доступа (allow и deny).

## 6. Безопасность и ПД (152-ФЗ, данные детей)
- Минимизация ПД: не собираем и не логируем лишнего о несовершеннолетних.
- Никаких секретов в коде/репозитории — только переменные окружения.
- Проверки прав — на бэкенде, не доверять фронтенду. Каждый доступ к данным ученика
  проверяется по матрице прав из спеки.
- Внешний ввод валидируется. Никаких сырых SQL-строк из ввода.
- ОГРАНИЧЕНИЕ: ИИ-ревью (А5) не заменяет внешний аудит безопасности перед публичным
  запуском с реальными данными детей.

## 7. Границы режима
- Пишешь код строго по спеке. Нет спеки/неоднозначно → стоп, к Архитектору.
- Продуктовые и стратегические решения — не твои, это фаундер.
- Тексты интерфейса — зона А8, не выдумывай их в коде как финальные.
- Контракты данных не меняешь без обновления спеки (см. §3).

## 8. Definition of Done
Фича готова, когда: код по спеке; контракты не нарушены; тесты написаны и зелёные;
CI зелёный; нет секретов в коде; проверки прав на месте; диф готов к ревью А5.
Следующий шаг после готовности дифа — отдельный чат с А5 (ревьюер кода) + спека + этот файл.

## 9. Экономия токенов (без потери качества)

### 9.1 Модель по задаче
Не всё требует Opus. Переключай `/model` по задаче:
- **Sonnet (по умолчанию):** рутинный код, правки строк, коммиты, brain-дельты, А8 (копирайт),
  простые А5-ревью, обновление документов, тесты. Это ~80% работы.
- **Opus (только когда нужно):** А3 (архитектура), А4 (критика системы), сложные А5
  (первый код нового модуля, High-risk по классификации), продуктовые разборы А9.

Перед запуском субагентов А3, А4 или сложного А5 — напомни фаундеру: «Рекомендую переключить на Opus (/model) для этой задачи». После завершения — напомни переключить обратно на Sonnet.

### 9.2 Конвейер по уровню риска
Полный конвейер (А3 → А4 → validator → код → тесты → А5 → апрув фаундера)
запускать только на **High-risk** задачах: новая модель данных, новая роль/права,
новый FSM, новый HTTP-контракт.

Сокращённый конвейер (код → тесты зелёные → коммит):
**Low-risk** — правка текста, обновление документа, добавление теста, brain-дельта,
правка одной строки в i18n, косметический рефакторинг.

Промежуточный (код → тесты → А5 → коммит, без А3/А4):
**Mid-risk** — новый компонент по принятой спеке, новый endpoint по принятому
API-аддендуму, изменение существующей логики в рамках спеки.

### 9.3 Контекст = токены
- **Новая сессия после каждого закоммиченного пункта.** Чем длиннее контекст,
  тем больше токенов на каждое следующее сообщение. Claude Code подхватит
  состояние из файлов на диске, не из истории чата.
- **Не копируй куски спек в задачу.** Достаточно ссылки: «по specs/student_lesson_api_v1.md, §3.1».
  Claude Code сам прочитает файл — это дешевле, чем передать текст в сообщении.
- **Не повторяй инструкции, которые уже в CLAUDE.md.** Этот файл читается
  автоматически — дублировать «отвечай на русском» или «стоп после каждого пункта»
  в каждом сообщении = лишние токены.

### 9.4 Субагенты экономнее отдельных чатов
Субагент в `.claude/agents/` получает **изолированный** контекст (только то, что
ему передано + файлы, которые он сам прочитает). Это дешевле, чем отдельный чат
в Claude.ai, где в контекст попадает весь проект. Для А8 (копирайт), А5 (ревью),
А9 (разбор развилки) предпочитай субагентов Claude Code, а не отдельные чаты.
Отдельные чаты — только для А0 Lead (ритуал, свежий контекст обязателен)
и А6 (UI-каркас с визуалом, если нужны mockup'ы).

## 10. Совместная работа Claude Code и Codex

### 10.1. Назначение
В проекте используется разделение труда между двумя инструментами:

- **Claude Code** — основной контур архитектуры и длинной реализации.
- **Codex** — внешний контур точечной инженерной работы, ревью и git-операций.

Цель такого разделения — экономить токены в Claude Code без потери качества
разработки. Между инструментами передаётся НЕ история чата, а короткий
формальный handoff-артефакт.

### 10.2. Где лежит протокол
Подробный протокол работы и маршрутизация задач лежат в:

- `docs/Протокол_ClaudeCode_Codex_v2.md`
- `docs/Маршрутизация_ClaudeCode_Codex_v2.md`

Шаблоны передачи лежат в:

- `prompts/Шаблон_передачи_в_Codex_v2.md`
- `prompts/Шаблон_возврата_из_Codex_v2.md`

Перед выполнением handoff-задачи Claude Code обязан опираться на эти файлы, а
не импровизировать произвольный формат.

### 10.3. Что остаётся в Claude Code
Claude Code ведёт:

- A3 Архитектора;
- A4 Критика системы;
- задачи уровня High-risk;
- новое поведение без принятой спеки;
- изменения контрактов CSV / FSM / API / permissions / data model;
- длинные многошаговые реализации;
- обновления Brain / Methodology / мастер-правил проекта;
- архитектурные развилки до начала кода.

### 10.4. Что уходит в Codex
Codex используется для:

- A5-style code review;
- проверки фактического состояния репозитория;
- git-операций и синхронизации GitHub / GitLab;
- работы с ветками, remotes, push/fetch/merge/compare;
- узких Mid-risk задач по уже принятой спеке;
- Low-risk исправлений;
- независимой инженерной проверки перед возвратом в Claude Code.

### 10.5. Обязательные триггеры передачи в Codex
Claude Code обязан остановиться и выдать handoff в Codex, если выполнено хотя бы одно условие:

1. Задача дошла до стадии ревью кода или дифа.
2. Следующий шаг — git-операции или синхронизация:
   fetch, pull, compare branches, push, remote-check, GitHub/GitLab alignment.
3. Спека уже принята, и оставшаяся работа узкая и локальная.
4. Нужно проверить реальное состояние файлов, веток, remotes или тестов.
5. Нужна независимая вторая пара глаз.
6. Claude Code упирается в лимиты, а оставшаяся работа не требует нового архитектурного рассуждения.

### 10.6. Когда передавать в Codex нельзя
Claude Code не должен передавать задачу в Codex, если:

- поведение новое и спеки ещё нет;
- меняются контракты данных;
- проектируются права доступа, ПД или приватность;
- создаётся новая роль или новый FSM;
- есть неразрешённая продуктовая развилка;
- требуется обновление Brain / Methodology;
- архитектурная неоднозначность ещё не снята.

### 10.7. Формат остановки
При срабатывании триггера Claude Code не продолжает работу "по инерции", а
останавливается и печатает:

`STOP: задача для Codex`

Далее Claude Code обязан выдать handoff СТРОГО по шаблону
`prompts/Шаблон_передачи_в_Codex_v2.md`.

### 10.8. Формат возврата
После выполнения задачи в Codex фаундер возвращает результат в Claude Code по
шаблону `prompts/Шаблон_возврата_из_Codex_v2.md`.

Claude Code обязан принимать такой возврат как рабочий входной артефакт и
продолжать следующую стадию задачи без требования пересказывать весь предыдущий чат.

### 10.9. Язык
Язык взаимодействия между фаундером, Claude Code и Codex — русский всегда.

На русском оформляются:
- протоколы;
- handoff-шаблоны;
- инструкции;
- отчёты;
- результаты ревью;
- маршрутизация задач;
- рабочие пояснения для фаундера.

На английском остаются только инженерные артефакты стандарта разработки:
- commit messages;
- имена переменных;
- комментарии в коде;
- identifiers;
- технические имена API и структур, если они нужны коду.

### 10.10. Правило сомнения
Если есть сомнение, куда нести задачу:
- вопрос архитектуры -> остаётся в Claude Code;
- вопрос проверки / ревью / sync -> уходит в Codex.
