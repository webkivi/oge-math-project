## ЭТАП 0 — Старт. Просто чат с Claude
**Период:** начало 2026 (приблизительно)
**Инструменты:** Claude.ai web, чат без проектов и агентов

### Решение 0.1 — Определить первый продукт
**Контекст:** Идея делать образовательный продукт для подготовки к ОГЭ/ЕГЭ. Много вариантов: какой предмет, какие задания, какой формат.
**Решение:** Один продукт глубоко — ОГЭ математика, задания №1–5 (~28 уроков). Не десять поверхностно.
**Причина:** Ограниченные ресурсы (5–15 ч/нед, бюджет до 5000 ₽/мес). Лучше один отличный продукт, чем пять средних.
**Результат:** Чёткий фокус, который держится до сих пор.

### Решение 0.2 — Выбрать канал: Telegram-бот через Salebot
**Контекст:** Нужен канал доставки уроков ученикам. Варианты: сайт, приложение, Telegram, ВКонтакте.
**Решение:** Telegram-бот на конструкторе Salebot (~1500 ₽/мес).
**Причина:** Школьники уже в Telegram. Конструктор — без своей разработки. Быстрый старт. Бюджетно.
**Результат:** Первая архитектура: контент в Google Docs → заливается в бот вручную. Работает, но не масштабируется.
**Ограничения которые вскрылись:** Адаптивные интервалы повторений в Salebot не реализуются (фиксированные таймеры). Кастомная логика сложная. База пользователей «внутри Salebot» — риск.

### Решение 0.3 — Гибридная модель производства контента
**Решение:** Фрилансер делает технический каркас бота (39–47 ч), фаундер делает контент. Бюджет на фрилансера: 8–12 тыс. ₽ единоразово.
**Причина:** Технический каркас — не компетенция фаундера. Контент — только фаундер знает, как должно быть.

## ЭТАП 1 — Структура. Появляются Методология и Project Brain
**Период:** апрель–май 2026
**Инструменты:** Claude.ai Projects (чат с документами проекта)

### Решение 1.1 — Создать два управляющих документа
**Контекст:** Сессии с Claude каждый раз начинались с нуля. Контекст терялся. Решения противоречили друг другу.
**Решение:** Два документа-«конституции»:
- `Methodology` — педагогические принципы, правила контента, стандарты качества (Методология v1.0 → сейчас v2.1)
- `Project Brain` — стратегия, статус, журнал решений (Brain v1.0 → сейчас v3.2)
**Причина:** Нужна единая точка истины. Claude должен работать в рамках зафиксированных принципов, а не изобретать заново каждый раз.
**Результат:** Все последующие решения документируются. Новую сессию можно начать с загрузки этих файлов — и Claude сразу в контексте.

### Решение 1.2 — Зафиксировать педагогические принципы в Методологии
**Ключевые решения зафиксированные в Методологии:**
- 6-этапная структура каждого урока (неизменна)
- 8-интервальное интервальное повторение
- Голос бота — «старший друг», «ты», без формализма
- Запрет тёмных паттернов: нет давящих счётчиков, нет наказания за ошибки, нет сравнения с другими
- Слово «стрик» — запрещено в UI (школьники не понимают)
- LLM не используется в живом чате с учеником (риск репутации)
**Причина:** Без зафиксированных принципов каждый агент (и каждая сессия) будет изобретать по-своему.

### Решение 1.3 — Перейти с Salebot на собственную разработку (PWA)
**Контекст:** Salebot работал как канал доставки, но блокировал: нельзя реализовать адаптивные повторения, кастомную логику, нормальный UX.
**Решение:** Переход на PWA (React + Vite + Tailwind) с Python-бэкендом.
**Причина:** Адаптивность требует собственной логики. Данные должны жить у нас, не в конструкторе. Salebot — потолок.
**Компромисс принятый сейчас:** Адаптивные интервалы повторения (SuperMemo-алгоритм) — отложены на v3.0/2027. Сначала фиксированные + ручная логика, потом автоматика.
**Результат:** Начата разработка PWA. Появились FSM-спецификации (reg_v2, lesson_v4).

## ЭТАП 2 — Агентная система. Команда AI-агентов
**Период:** май–июнь 2026
**Инструменты:** Claude.ai Projects + файлы агентов (.md промпты)

### Решение 2.1 — Создать систему пронумерованных агентов А0–А9
**Контекст:** Разные задачи требуют разных «ролей» Claude. Каждый раз объяснять роль заново — потеря времени и качества.
**Решение:** Каждый агент = отдельный .md файл с системным промптом, ролью, зонами ответственности и ограничениями.
**Архитектура ролей:**
- А0 — Координатор / аудит согласованности (не создаёт, только проверяет)
- А1 — Контент (учебные материалы)
- А2–А5 — архитектура, педагогика, аналитика, доступность
- А6 — UI/дизайн-система
- А7 — разработка (→ Claude Code)
- А8 — копирайтер (микрокопи)
- А9 — тестирование

**Причина:** Разделение ответственности. Каждый агент знает что его, а что — чужое. А6 не пишет тексты — это А8. А8 не проектирует экраны — это А6.
**Результат:** Чёткий workflow: A6 делает дизайн-систему с плейсхолдерами `[A8: …]` → A8 закрывает их микрокопи → Claude Code реализует.

### Решение 2.2 — Ввести ритуал аудита (Агент А0)
**Контекст:** После нескольких итераций документы начали расходиться: версии в заголовках устарели, статусы агентов не обновлены, закрытые решения висели как открытые.
**Решение:** А0 — специальный аудитор. Запускается отдельно, только читает все артефакты и выдаёт таблицу дрейфа.
**Первый аудит нашёл:** 13 drift-позиций: устаревшие версии в заголовках, А6–А9 числились «не созданы» хотя уже существовали, Telegram-механика в §3.7 без пометки что это иллюстрация будущего, а не текущее.
**Результат:** Перед каждой новой волной работы — ритуал А0. Иначе агенты начнут работать на устаревших данных.

### Решение 2.3 — Иерархия приоритетов документов
**Решение:** Жёсткая иерархия: Методология > Project Brain > промпты агентов > артефакты агентов.
**Причина:** Если агент видит конфликт между своим промптом и Методологией — Методология побеждает всегда. Это защита от дрейфа.

## ЭТАП 3 — Переход в Claude Code
**Период:** июнь 2026
**Инструменты:** Claude Code (CLI / Desktop) + чат Claude для стратегии

### Решение 3.1 — Подключить Claude Code для технических задач
**Контекст:** Реализация PWA требует писать и редактировать код, настраивать сервер, работать с файлами. Чат-интерфейс для этого неудобен.
**Решение:** Claude Code для технических задач (разработка, деплой, конфиги). Чат — для стратегии, контента, дизайна.
**Конфигурация:** `CLAUDE.md` в корне проекта автоматически загружает контекст (Methodology + Brain) в каждую сессию.

### Решение 3.2 — Выбор серверной архитектуры
**Решение:** VPS (Timeweb Cloud / Selectel) + Ubuntu 24.04 + nginx + PostgreSQL + systemd.
**Причина:** Рублёвая оплата. Полный контроль. Стандартный стек, который Claude Code хорошо понимает.
**Схема:** Браузер → nginx (443) → статика (фронтенд) / proxy_pass → Python-бэкенд (localhost).

### Решение 3.3 — Режимы автоматизации Claude Code
**Контекст:** Нужно запускать агентные задачи без постоянного ручного подтверждения.
**Решение:** `acceptEdits` для автоподтверждения правок файлов. `--dangerously-skip-permissions` для headless-режима (изолированно, в контейнере).
**Ограничение:** Headless-сессии иногда зависают на permission-промптах — только bypass-режим решает.

## ЭТАП 4 — Текущее состояние (июнь 2026)
**Инструменты:** Claude.ai чат + Claude Code (подключается для технических задач)

### Статус продукта
- PWA в разработке: React/Vite/Tailwind
- FSM-спецификации: reg_v2 (регистрация), lesson_v4 (урок) — готовы
- Дизайн-система: `A6_UI_дизайн-система_ученик_v1.md` — готова (WCAG 2.1 AA)
- Микрокопи: `A8_microcopy_ученик_v1.md` — 33 строки готовы
- Открытые фаундерские решения: F-1 (видимость счётчика дней), F-2 (редактируемость ника/класса)

### Открытые архитектурные вопросы (из аудита А0)
- Telegram/§3.7 — нужна пометка что это иллюстрация, не текущая реализация
- Статус волнового плана (все роли → потом реализация vs. параллельно)

### Решение 4.1 — Закрыть вертикальный срез «прохождение урока» живым прогоном
**Дата:** 28.06.2026
**Контекст:** Backend и frontend урока были реализованы и приняты ревью А5, но без проверки вживую — оставался риск, что спека и реальное поведение расходятся (как уже было с регистрацией). При живом прогоне нашёлся реальный баг: после выхода из урока (`evt_cancel_lesson`) повторный вход начинал урок заново, а не с сохранённой позиции (EC-02), хотя по спеке прогресс должен сохраняться.
**Решение:** Не считать срез готовым по одному прохождению тестов — обязательный шаг: живой прогон критических сценариев (wrong+return, mastery 2-я попытка, провал, cancel+resume) через реальный API/браузер. Баг EC-02 исправлен (`_resume_state()` в `fsm_service.py`), прошёл два круга независимого ревью через Codex.
**Причина:** Юнит-тесты проверяли отдельные переходы FSM, но не полный round-trip «cancel → повторный вход» — именно там пряталась дыра. Тесты, написанные после факта, теперь закрывают именно этот сценарий.
**Результат:** Вертикальный срез урока подтверждён end-to-end вживую на уроках 1_1–1_8, бэкенд-бага закрыт, 249 backend-тестов зелёные.
**Инструменты:** Claude Code (WSL2 + uvicorn + vite), Codex (git-операции и повторное ревью кода через handoff-протокол).

### Решение 4.2 — Закрыть легаси-долг контента Блока 1 (1.3–1.9) тем же конвейером, что и новые уроки
**Дата:** 28–29.06.2026
**Контекст:** 9 уроков Блока 1 были написаны до того, как правило keeper.py §3.3 (распределение букв) заработало в коде — урок 1.9 не проходил keeper вовсе, а 1.3–1.8 формально проходили, но никогда не использовали буквы C/D как верный ответ (легаси, годами числившийся в Brain как «в очереди на Критика»).
**Решение:** Прогнать все 7 проблемных уроков через штатный конвейер Продюсер→keeper.py→изолированный Критик (без памяти о ходе правок других уроков), строго по одному уроку за раз с приёмкой фаундера между уроками — не пакетно и не параллельно, чтобы не размыть независимость критика.
**Причина:** «Автор не судит себя сам» — несущий принцип проекта. Пропустить эту дисциплину для «мелкого» исправления легаси означало бы тихо ослабить процесс именно там, где он самый ценный — на старом контенте, который никто давно не перечитывал.
**Результат:** Блок 1 (1.1–1.9, 9 уроков) полностью закрыт контентно: keeper.py PASS и вердикт GO Критика на всех. По пути конвейер поймал реальные находки (ритуал метода вместо затухания в 1.3, неоднозначные дистракторы, повторяющийся тип ошибки) и одну ложную эскалацию (код ФИПИ урока 1.8), закрытую решением фаундера.
**Инструменты:** Claude Code как оркестратор + субагенты `producer`/`lesson-critic`, keeper.py.

### Решение 4.3 — Первый продакшен-деплой на VPS (team-management.ru)
**Дата:** 29–30.06.2026
**Контекст:** Бэкенд и фронтенд были готовы и протестированы локально (Решение 4.1), но никогда не разворачивались на реальном сервере. `main.py` уже год как декларировал намерение «в production схему ведёт Alembic», но миграций физически не существовало — разрыв между обещанием в коде и инфраструктурой.
**Решение:** Развернуть полный стек на VPS Timeweb (Ubuntu 24.04): PostgreSQL вместо SQLite, Alembic-миграции (созданы с нуля в этой сессии — `db/migrations/`), backend как systemd-сервис, nginx как reverse-proxy + статика фронтенда, SSL через certbot. Доступ к серверу — только через браузерную консоль Timeweb (SSH по паролю с внешних IP заблокирован), весь процесс — пошаговая передача команд фаундеру для ручного выполнения.
**Причина:** «Версии фиксируются при первом деплое (DevOps, А7)» — комментарий в `requirements.txt` явно делегировал эту работу А7, а не Архитектору; это инфраструктурная задача по уже принятой спеке данных, не новый FSM/контракт.
**Находка по пути:** DNS A-запись домена на reg.ru указывала на чужой IP (`31.31.196.74` вместо реального `217.25.93.57`) — обнаружено сравнением «что видит сервер локально» против «что видно снаружи» (curl с разных сторон давал разный контент). Без этой проверки certbot продолжал бы падать без понятной причины.
**Результат:** Сайт живой на `https://team-management.ru`, end-to-end проверено вручную в браузере (регистрация → запись подтверждена в PostgreSQL). Добавлены резервное копирование БД (ежедневно, хранение 14 дней), ротация логов journald, watchdog с автоперезапуском каждые 5 минут.
**Инструменты:** Claude Code (генерация команд + анализ кода) + браузерная консоль Timeweb (ручное исполнение фаундером) + Claude in Chrome (E2E-проверка регистрации) + DoH-запросы к публичным DNS-резолверам (диагностика, в обход искажённого DNS в собственном sandbox-окружении).

### Решение 4.4 — Закрыть тупиковый «пустой экран дня» содержательным render-контрактом без правки FSM
**Дата:** 30.06.2026
**Контекст:** После живого прогона урока (4.1) и продакшен-деплоя (4.3) стало видно, что вертикальный срез успешного дня заканчивается пустым экраном «Всё на сегодня закончено» — методический и продуктовый тупик, который ученик увидит сразу после первого же урока. Нужен экран «День N завершён» с прогрессом курса, итогом дня, превью завтрашнего урока, streak-фидбэком.
**Решение:** Принята спека `specs/day_end_screen_v1.md` (v1 + R1 + R2). Расширение render-payload api v1 §4.1 одним опциональным блоком `day_summary` для трёх существующих view (`day_done`/`day_blocked`/`course_complete`) — БЕЗ правок FSM v4 и без правок матрицы прав v4 §3. Новые поля: `day_number`, `day_recap`, `course_progress` (с опц. `current_block`), `next_lesson_preview?` (опц. на `day_done`), `streak_today`, `push_cta?`, `block_completed_today?`, `course_completion?` (для финала курса), `system_paused?` (диагностический флаг рассинхрона манифеста). Контракт самой push-подписки вынесен в отдельную будущую спеку `push_subscription_api_v1`.
**Причина:** FSM v4 уже различает 3 семантически разных терминала дня (`daily_done`, `daily_blocked`, `course_complete`) с разными guards, выходами и правами-следствиями — слияние их в одно `day_end`-состояние сломало бы v4 §2б и матрицу прав без выигрыша. Дефицит был в render-payload, а не в автомате. Расширение через опциональное поле верхнего уровня сохраняет обратную совместимость с принятым api v1.
**Результат:** Спека прошла 3 независимых раунда Критика (А4) с финальным чистым GO. На приёмке фаундера остались 6 продуктовых развилок (DE-1..DE-6: поздравление при конце блока, прогноз ОГЭ + ПД-классификация, политика push, превью на `day_done`, `longest_streak`, декомпозиция курса на блоки) и 4 правки контракта данных (`LESSON_TITLES` — обязательно, `COURSE_BLOCKS` — опционально, `StudentProfile.tz`, F-12 «course manifest desync»). Реализация — Mid-risk с UI (код → тесты → А5 → А1О UX-Gate) после закрытия предусловий, включая зависящие открытые правки api v1 (§3.4 `COURSE_MANIFEST`, §3.4-bis единое пространство `lesson_id`).
**Инструменты:** Claude Code (А3 Архитектор, 3 ревизии спеки) + субагенты `system-critic` (3 независимых раунда А4 ревью с изолированным контекстом каждый).
