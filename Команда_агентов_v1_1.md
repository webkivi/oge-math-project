## 1. Принципы команды

- **Автор не судит себя сам.** Любой значимый артефакт проходит через критика в отдельном чате, без накопленного контекста автора.

- **Детерминированный шлюз там, где можно.** Если ошибку ловит скрипт (keeper.py, validator.py, автотесты) — её ловит скрипт, не агент. Критик-человекоподобный судит только то, что нельзя свести к проверке.

- **Минимум персон, не максимум.** Каждая персона-промпт — отдельная дисциплина запуска и точка рассинхронизации. Создаём только тех, кто закрывает реальный риск. Процедуры (миграции, инцидент-ответ) и режимы (разработка) — это не персоны.

- **Каждая роль — одно дело.** Никаких «универсальных солдат», которые и пишут, и ревьюят, и архитектурят в одном чате.

- **Нагрузка на фаундера ограничена классификацией риска.** Полный конвейер из 6 стадий — только для опасных изменений. Большинство задач идут коротким путём (см. раздел 4).

## 2. Что такое «персона», «шлюз», «режим», «процедура», «ритуал»

| **Тип** | **Что это** | **Где живёт** |
| --- | --- | --- |
| Персона | Промпт с фиксированной ролью, запускается в чате на конкретную задачу | .txt шаблон в проекте |
| Шлюз | Детерминированный скрипт, даёт PASS/FAIL | Python в репозитории |
| Режим | Набор правил, по которым Claude работает в инструменте | CLAUDE.md в репозитории |
| Процедура | Чек-лист + промпт под повторяющуюся операцию | .txt + .md в проекте |
| Ритуал | Периодический запуск фаундером в свежем чате | .txt шаблон, запуск по расписанию |

## 4. Роли подробно

### А1. Продюсер уроков

**Дело:** генерирует урок по промпту Агент_1_Продюсер_v4.1 — один файл .csv (машинный источник, его читает движок) + чек-лист. **Вход:** номер урока, Methodology v2.1, Project Brain v3.2, ФИПИ. **Выход:** один CSV (Контент_урок_N_N.csv) + самопроверка по чек-листу; вычитка фаундером ведётся по CSV, отдельный .docx не производится. **Эталон:** формат — контракт CSV (keeper.py, универсально); структура — архетип блока (блок 1 равняется на урок 1.1, блоки 2+ — по структурному брифу блока). Мастер-промпт слит в А1. **Где:** отдельный чат на каждый урок. **Кто проверяет:** А2 + keeper.py. **Статус:** работает (v4.1).

### А2. Критик уроков

**Дело:** рвёт урок по рубрике A–K, независимо пересчитывает арифметику в Python, сверяет с ФИПИ. Вердикт GO / правки / NO-GO. **Вход:** урок от А1, Methodology, Brain, ФИПИ. Видит только это — не видит рассуждения Продюсера. **Где:** отдельный чат, никогда не тот же, где работал А1. **Статус:** работает.

### keeper.py

**Дело:** детерминированный шлюз на CSV — байты (BOM/CRLF/19 колонок), парность HTML-тегов, эмодзи-листы, корректность correctanswer и return, обязательные stage, отсутствие мнемоники. **Выход:** PASS / список ошибок. **Статус:** работает.

### А3. Архитектор системы

**Дело:** производит формальное описание поведения системы — словарь сущностей, конечный автомат на каждую роль (состояния + таблица переходов), матрицу прав (роль × ресурс × операция), межролевые сценарии, список edge cases с явным ожидаемым поведением, режимы отказа. **Вход:** задача (например «логика роли ученик»), Methodology, Brain, предыдущие артефакты архитектуры. **Выход:** артефакты в фиксированном формате (Markdown-таблицы + Mermaid FSM + YAML для validator). **Где:** отдельный чат на каждую задачу, новый каждый раз. **Кто проверяет:** А4 + validator.py. **Ограничение:** Архитектор проектирует на бумаге; реальные дыры всплывут на пилоте. «Прошло Архитектора» ≠ «безошибочно». **Статус:** работает (v1).

### А4. Критик системы

**Дело:** рвёт спецификацию Архитектора по рубрике: полнота сущностей, полнота FSM (каждое состояние × событие обработано), достижимость, тупики, консистентность прав, приватность ученика, 152-ФЗ, UX-логика последовательности, идемпотентность, обратимость, соответствие Methodology. Вердикт GO / GO WITH REVISIONS / NO-GO. **Вход:** одна версия артефакта Архитектора + Methodology + Brain. Не видит историю итераций. **Где:** отдельный чат, не тот, где работал А3. **Ограничение:** см. раздел 6 — кто проверяет проверяющего. **Статус:** работает (v1).

### validator.py

**Дело:** читает FSM как YAML и механически проверяет: у каждого состояния есть вход и выход (кроме start/end); каждое событие обработано; BFS от старта достигает всех состояний, кроме явно помеченных недостижимыми; матрица прав без NULL; нет противоречий между сценариями. Читает FSM-YAML по контракту А3 (states/events/transitions/permissions); контракт меняется только связкой А3→А4→validator. **Выход:** PASS / список нарушений. **Статус:** работает.

### Режим разработки (Claude в Claude Code)

**Дело:** написание кода по спецификации Архитектора — бэкенд FastAPI, фронтенд React+Tailwind, channel-адаптеры (TG/MAX/VK), PWA, миграции БД. **Тип:** это НЕ персона-промпт, а режим работы Claude в инструменте Claude Code. Правила режима лежат в CLAUDE.md в корне репозитория и читаются Claude в начале каждой сессии: фиксированные конвенции кода, «читай Methodology и спецификацию Архитектора перед началом», «не меняй контракты данных без обновления спецификации», «каждая фича — с тестами». **Почему режим, а не персона:** Claude Code стартует с чистого контекста каждый раз; персона-личность там не сохраняется. Единственный надёжный носитель правил — файл в репозитории, который инструмент сам читает. **Где:** локальная среда фаундера (терминал, git, Node.js, Python). **Кто проверяет:** А5 + автотесты. **Статус:** работает (v1).

### А5. Ревьюер кода

**Дело:** ревьюит каждый значимый диф — соответствие спецификации Архитектора, читаемость, обработка ошибок, очевидные уязвимости, наличие тестов. **Вход:** диф (PR) + спецификация + CLAUDE.md. **Где:** отдельный чат с дифом. **Ограничение (явно):** это та же модель, что писала код, в свежем контексте. Ловит значительную долю багов, но НЕ заменяет внешний аудит безопасности. Уязвимости, невидимые обеим инстанциям, проскочат. Перед публичным запуском с реальными данными детей — отдельный внешний аудит обязателен. **Статус:** работает (v1).

### Автотесты + CI

**Дело:** pytest на бэкенд, vitest на фронтенд, smoke-тесты канал-адаптеров. Запуск на каждом коммите. **Выход:** PASS / FAIL по коммиту. **Статус:** создать.

### Процедура миграций данных

**Дело:** когда меняется структура CSV или схема БД — конвертация существующего состояния учеников без потери прогресса, с возможностью отката. **Тип:** процедура — чек-лист (бэкап до миграции, dry-run, проверка на копии БД, план отката) + промпт для генерации миграционного скрипта. Выполняется в режиме разработки. **Почему не персона:** это редкая операция по фиксированному чек-листу, не творческая роль. **Статус:** создать (чек-лист + шаблон).

### А6. UI-каркас и дизайн-система

**Дело:** компетентные дефолтные интерфейсы — wireframes экранов PWA, дизайн-система (палитра, шрифты, отступы, компоненты), пользовательские потоки, начальный microcopy-каркас (финальный текст — у А8). **Вход:** спецификация ролей от Архитектора, принципы Methodology («15 минут», «один экран — одна мысль»). **Выход:** дизайн-токены, mockup в HTML/SVG, карта экранов. **Ограничение (явно и важно):** это «компетентные дефолты», НЕ настоящий UX. Настоящий UX — это наблюдение за реальными учениками, итерации на данных, эстетическое суждение из опыта с этой аудиторией. ИИ-агент этого не заменяет. Полноценный UX-цикл (с UX-критиком) стартует, когда есть 5+ живых учеников для тестирования. До этого момента интерфейс не имитируем как «выверенный» — он «разумный по умолчанию, ждёт проверки людьми». **Статус:** работает (v1).

### А7. DevOps-инженер

**Дело:** Docker compose, деплой-инструкции, мониторинг (healthcheck + внешний аптайм-чек), бэкапы БД с ротацией, авто-перезапуск контейнера, SSL-renewal, runbook «что нажать, если упало». **Вход:** код от режима разработки, требования к надёжности из Brain §5.6. **Выход:** конфигурации + runbook + команды. **Кто проверяет:** ежемесячный chaos-тест фаундера (убить контейнер — проверить авто-restart). Отдельного критика на старте нет — оправдано масштабом. **Статус:** работает (v1).

### Процедура инцидент-ответа

**Дело:** бот упал — что делает фаундер. Runbook с конкретными шагами + промпт «диагностика по логам», который не требует переоткрывать весь контекст проекта. **Тип:** процедура (часть вертикали эксплуатации). Запускается фаундером в момент инцидента. **Почему**** ****отдельно от DevOps:** DevOps строит инфраструктуру заранее; инцидент-ответ — это реакция в моменте, с другим входом (логи, а не требования) и другой срочностью. **Статус:** создать (runbook + промпт).

### А8. Копирайтер продуктового текста

**Дело:** тексты онбординга, регистрации, кнопок, сообщений об ошибках, описаний ролей в PWA, уведомлений. Это НЕ контент уроков (зона А1) и НЕ маркетинг (зона фаундера) — это «голос интерфейса». **Вход:** экраны от А6, тон из Methodology 3.1 («старший товарищ, который шарит»). **Выход:** тексты строками, готовые к вставке в код/CSV. **Кто**** ****проверяет:** фаундер (голос продукта — его зона) + сверка тона с Methodology может делаться на ритуале Lead. **Статус:** работает (v1).

### А0. Project Lead (Хранитель согласованности)

**Дело:** следит, чтобы Methodology / Brain / промпты агентов / спецификации не противоречили друг другу; ловит дрейф (решение записано в одном документе, но не доехало до других); обновляет журнал решений; предлагает, кто работает следующим. **Тип:** РИТУАЛ, не постоянная роль в основном чате. Фаундер раз в 1–2 недели открывает новый чат с промптом Lead и набором актуальных артефактов; агент сверяет их со свежим контекстом, без накопленной истории. **Почему ритуал, а не роль в**** ****рабочем чате:** Lead внутри основного рабочего чата — это автор, судящий себя. Накопленный контекст ослепляет к собственному дрейфу. Свежий чат — единственный способ получить настоящую внешнюю сверку. **Кто проверяет Lead:** фаундер. Это единственная роль без агента-пары — её критикует человек. **Статус:** работает (v1).

### А9. Продуктолог (UX × платформа)

**Дело:** разбирает один открытый продуктовый/UX-вопрос (развилку), взвешивает с двух объективов — польза ученику и интерес платформы — и даёт обоснованную рекомендацию. **Не решает** и не пишет код/схему/контент: решение за фаундером, «как» — за Архитектором (А3). **Приоритет интересов:** польза ученику > удобство вспомогательных ролей (родитель/учитель/репетитор) > интерес платформы; красная линия — без тёмных паттернов, продукт для детей. **Вход:** один вопрос, Methodology, Brain. **Выход:** разбор (суть развилки → варианты → конфликт интересов → рекомендация → что нужно от фаундера). **Где:** отдельный чат. **Место в конвейере:** продуктовый вопрос → А9 → фаундер решает → А3 формализует «как». **Статус:** работает (v1.1).

## 5. Классификация работ по риску

Чтобы команда не задушила себя шестистадийным конвейером на каждую мелочь, каждая задача классифицируется до начала:

| **Уровень** | **Что сюда входит** | **Конвейер** |
| --- | --- | --- |
| High-risk | Новая роль; новая модель данных; изменение прав доступа; изменение методологии; работа с ПД | Архитектор → Критик системы → validator → Разработка → Ревью кода → автотесты |
| Mid-risk | Новая фича на существующей модели, не трогающая права и ПД | Архитектор (кратко) → Разработка → автотесты |
| Low-risk | Косметика, продуктовый текст, мелкий багфикс, рефакторинг без смены контрактов | Разработка → автотесты |

Уровень определяет фаундер вместе с Lead на старте задачи. При сомнении — берём уровень выше.

## 6. Явные ограничения ИИ-команды

Зафиксировано, чтобы не возникало ложной уверенности «прошло агентов — значит безупречно»:

- **ИИ-ревью не заменяет внешний аудит безопасности.** Код пишет и ревьюит одна модель. Перед публичным запуском с данными детей нужен независимый аудит.

- **ИИ-агент не заменяет настоящий UX.** До появления живых тестировщиков интерфейс — «разумный дефолт», не «выверенный продукт».

- **Кто проверяет проверяющих.** Если Критик системы (А4) пропустит дыру — её поймает только реальность на пилоте. Ритуал Lead снижает риск, но не устраняет.

- **Самокритика автора — нижняя оценка проблем, не верхняя.** Независимый критик в свежем чате находит больше. Поэтому отдельные критики обязательны, а не «достаточно подумать ещё раз».

- **152-ФЗ — зона фаундера.** Агенты проектируют систему так, чтобы минимизировать сбор ПД, но юридическую ответственность оператора несёт фаундер.
