- Куда идём (стратегия)

1.1 Видение

Платформа адаптивного микро-обучения, начиная с ОГЭ по математике. Конкурентное преимущество — доходимость 60%+ против рыночных 20–30%. Платформа растёт по двум осям: удобство (для ученика, родителя, учителя, репетитора) и доказанная эффективность методики. Эти две оси — единственные ограничители развития; календарь и смета продукт не лимитируют. Базовая поверхность — PWA (веб-приложение, работает у всех без установки из магазина); мессенджеры подключаются позже как дополнительные поверхности для ученика.

1.2 Определение успеха

Успех — это доказанная методика, а не выручка. Три компонента:

- ≥ 800 учеников, начавших хотя бы один курс.

- ≥ 60% доходимости до конца на когорте 100+ человек.

- 3–5 задокументированных кейсов «был X — стал Y» с согласием родителей.

Дат у этой цели нет: она достигается тогда, когда методика и платформа к ней готовы.

1.3 Принципы

- Польза ученику первична. Родитель и учитель обслуживают этот результат, а не конкурируют с ним за внимание.

- Один продукт глубоко, не десять поверхностно. На старте — ОГЭ математика, целевой результат «тройка».

- Удобство платформы и эффективность методики — главные критерии любого решения.

- Своими руками через код. Платформа пишется кодом (PWA + бэкенд); Claude реализует через Claude Code, фаундер владеет стратегией, контентом и приёмкой.

- PWA-first. Сначала одна поверхность — веб-приложение для всех ролей; мессенджеры добавляются переносом, не переписыванием.

- Технологии следуют за продуктом, а не наоборот.

- Если меняем принцип — фиксируем в Журнале решений.

- Что делаем (продукт)

2.1 Аудитории — четыре роли

Основная роль — ученик; он проходит уроки внутри PWA. Родитель, учитель и репетитор — вспомогательные роли, которые усиливают доходимость ученика и не вмешиваются в программу. Все четыре заложены в архитектуру с самого начала и работают в одном PWA (разные интерфейсы по типу аккаунта).

| **Роль** | **Что получает** | **Чего не может** |
| --- | --- | --- |
| Ученик (9 кл.) | Прохождение уроков в PWA, тренировки, повторения, счётчик дней, карта знаний | — |
| Родитель | Привязка к ученику по коду, еженедельная сводка прогресса, алёрт при долгом пропуске | Писать ученику от лица системы, менять программу, видеть содержимое уроков |
| Учитель | Класс по коду доступа, дашборд по классу и ученикам, рассылка объявлений классу | Ставить оценки, редактировать контент, видеть личную работу ученика |
| Репетитор | Привязка к своим ученикам по коду (плоский список), детализация по каждому ученику | Оценки, редактирование контента, доступ к чужим ученикам, личная работа ученика |

Связь «взрослый ↔ ученик» — многие-ко-многим с типом (родитель / учитель / репетитор): у одного ученика одновременно могут быть мама-родитель, учитель и репетитор. Это родовая ситуация, не край.

2.2 Поверхности (PWA-first)

Один бэкенд обслуживает несколько поверхностей. Бизнес-логика (поток урока, состояния, повторения, напоминания, роли, права) отделена от поверхности: новая поверхность — это адаптер, контент и логика не трогаются.

- **PWA — базовая поверхность, строится первой, сразу для всех ролей.** Веб-приложение: открывается по ссылке в любом браузере, ставится «на экран Домой» без магазина приложений, поддерживает офлайн и push. Ученик проходит уроки здесь; родитель, учитель и репетитор работают здесь же.

- **Мессенджеры — этап переноса.** После того как PWA проверена, ученику добавляются альтернативные поверхности через адаптеры: Telegram, MAX, ВКонтакте. Тот же контент и логика, другой адаптер.

Почему так: на июнь 2026 мессенджеры в РФ нестабильны как единственный канал (MAX удалён из App Store, у iPhone нет push; Telegram у большинства требует VPN; VK стабилен, но как основная среда учёбы для подростка под вопросом). PWA доступна по ссылке независимо от этого и работает у всех ролей сразу. Несколько одновременных поверхностей на одном бэкенде — страховка от блокировки любой из них.

| **Порядок** Сначала — PWA для всех ролей (ученик проходит уроки, взрослые видят прогресс). Затем — перенос ученической поверхности на Telegram, MAX, ВКонтакте через адаптеры. Нативное приложение из магазинов — отложено (см. §11). |
| --- |

2.3 Архитектурные страховки (без vendor lock-in)

- База пользователей — в собственной БД, под нашим контролем. Ежедневный бэкап.

- Контент — в CSV в нашем репозитории. Движок читает CSV при старте. Источник истины — файлы, не платформа.

- Поверхности через адаптеры — добавление PWA → мессенджеров не требует переписывания курса.

- Базовая поверхность (PWA) не зависит от магазинов приложений и от блокировок отдельных мессенджеров.

- Как работаем

3.1 Разделение труда

**Только Фаундер:** стратегия и продуктовые решения, общение с учениками/родителями/учителями, школы, финальная вычитка контента, юр.оформление, личный бренд. Аренда сервера и хранение секретов (токены ботов, доступы) — на стороне фаундера; Claude секреты не хранит.

**Claude:**

- Контент уроков через агентный конвейер (Продюсер → Критик → keeper.py).

- Код платформы через Claude Code: бэкенд (FastAPI), PWA-фронтенд (React/Vite/Tailwind), движок уроков, логика всех 4 ролей, адаптеры мессенджеров (этап переноса), напоминания, деплой-инструкции.

- Архитектура, документы, аналитика, дизайн логики, чертежи.

**Команда агентов** (полный реестр — в документе «Команда агентов v1.1»): контент (Продюсер, Критик уроков, keeper.py), архитектура (Архитектор, Критик системы, validator.py), код (режим разработки в CLAUDE.md, Ревьюер кода, автотесты), интерфейс (UI-каркас), эксплуатация (DevOps), продуктовый текст (Копирайтер), поперечный ритуал Project Lead. Работы делятся по риску High / Mid / Low — полный конвейер только для опасных изменений.

**Инструмент разработки:** Claude Code в локальной среде фаундера (Windows + WSL2 Ubuntu), Путь А.

**Фрилансеры:** не используются.

3.2 Конвейер производства урока (без изменений)

- Шаг 1. Агент 1 (Продюсер) генерирует урок по Мастер-промпту v4 — оба файла + чек-лист.

- Шаг 2. Агент 2 (Критик) в отдельном чате рвёт урок по рубрике, выдаёт вердикт + правки.

- Шаг 3. Продюсер вносит правки. Цикл до вердикта GO.

- Шаг 4. keeper.py (Хранитель) — детерминированный шлюз на CSV, должен дать PASS. (Это шлюз, не персона; «Агент 5» — это Ревьюер кода.)

- Шаг 5. Фаундер делает финальную вычитку и показывает живому 9-класснику (для первых 5 уроков обязательно).

- Шаг 6. Финальная правка. CSV кладётся в репозиторий — движок подхватывает его при перезапуске.

Ключевой принцип конвейера: автор не судит себя сам. Критик работает в отдельном чате и видит только Методологию + урок.

3.3 Дисциплина документов

Methodology (конституция) и Project Brain обновляются только осознанным решением. Claude предлагает дельту на ревью фаундеру до внесения любой правки. Десинхронизация (решение записано в методологию, но не доехало до контента/кода) — главный режим отказа проекта; против него работают keeper.py и единый источник истины в CSV.

- Методология (резюме; полная версия — в Methodology)

Доходимость — главная метрика. 60%+ против рыночных 20–30%. Опора — доказательная педагогика:

- **Mastery learning** — продвижение по освоению, а не по календарю.

- **Spaced repetition** — повторения через интервалы; в коде доступны адаптивные интервалы (ранее были заблокированы платформой).

- **Active recall** — ученик отвечает, а не просто читает.

- **Interleaved practice** — сюжеты вперемешку (61–77% против 38% на отсроченном тесте).

- **15 минут в день** — научный стандарт микро-обучения.

Метод чтения задачи подаётся через смысл 4 шагов (что дано → что спрашивают → как связать → проверка), без аббревиатур и кодовых имён: для слабого 9-классника код через кривую забывания вырождается в незнакомый символ.

Намеренно не учим: зад. 13, 14, 16, 17; теорему Виета; часть 2 (зад. 20–25). Это продуктовая фокусировка на тройке, а не ограничение.

- Архитектура кода

5.1 Стек

- **Базовая поверхность — PWA:** React + Vite + Tailwind, service worker (офлайн + push). Ученик проходит уроки на веб-экранах; родитель / учитель / репетитор работают в том же приложении с разными интерфейсами по роли.

- **Бэкенд:** Python + FastAPI. Вся бизнес-логика (поток урока как конечный автомат состояний, роли, права, повторения, напоминания) — здесь, отдельно от поверхности.

- **Хранилище:** SQLite на старте (бэкап — копирование файла). Переход на PostgreSQL — при росте, без переписывания логики (слой доступа изолирован).

- **Контент:** CSV в репозитории, та же структура (BOM + CRLF + 19 колонок). Движок читает CSV; рендерится как перенос на веб-экране.

- **Мессенджер-адаптеры (этап переноса):** Telegram (aiogram), MAX (REST/maxapi), ВКонтакте (Callback API). Каждый — тонкий адаптер над общим бэкендом; добавляются после валидации PWA.

- **Контейнеризация:** Docker Compose — поднять/перезапустить одной командой.

- **Доставка:** PWA по HTTPS; push через service worker. Мессенджеры — webhook/long polling на этапе переноса.

- **Мониторинг:** внешний аптайм-чек + самоуведомление фаундеру при сбое; ежедневный бэкап БД.

5.2 Модель данных (ядро)

- **users** — роль (student / parent / teacher / tutor), идентификатор аккаунта (PWA; для мессенджер-поверхностей — id в канале), имя, класс, дата старта.

- **progress** — ученик, урок, статус, попытки, ошибки, отметки времени.

- **streak** — счётчик дней подряд, передышки.

- **review_queue** — очередь тем, всплывающих на следующее утро (Вариант 3).

- **links** — связи «взрослый ↔ ученик», МНОГИЕ-КО-МНОГИМ, с полем тип (parent / teacher / tutor), по коду доступа. Один ученик может иметь несколько взрослых разных типов.

- **classes** — класс учителя, код доступа, список учеников.

- **reminders_state** — флаги триггеров напоминаний на пользователя.

Точная схема, состояния и матрица прав — артефакт Архитектора (Агент 3), проверяется Критиком системы (Агент 4) и validator.py до написания кода.

5.3 Логика ученика

- Ученик проходит урок на веб-экранах PWA: теория → пример → тренировка → главный вопрос → финал. Поток описан как конечный автомат состояний.

- Ветвление на неверных ответах (return_X) и обязательные stage берутся из CSV; маршрутизация return_X ↔ message_id решается в коде.

- Логика ошибки на закреплении — Вариант 3: переответ без откатов + всплытие темы на следующее утро через review_queue (поле в БД + утренняя проверка очереди).

- Напоминания: возвратные триггеры (пропуск сегодня / 2 дня / 5+ / 14+ / 30+) + сброс стрика. В PWA — через push; на мессенджер-поверхностях — через канал. Число триггеров не ограничено.

- Та же логика урока переиспользуется мессенджер-адаптерами на этапе переноса — без переписывания.

5.4 Логика родителя (в PWA)

- Привязка по короткому коду: ученик показывает код, родитель вводит его в своём боте.

- Еженедельная сводка: сколько уроков пройдено, серия дней, главные типы ошибок.

- Алёрт при пропуске 3+ дней.

- Жёсткие границы: родитель не пишет ученику от лица системы, не меняет программу, не видит содержимое уроков.

5.5 Логика учителя (в PWA)

- Создание класса и выдача кода доступа ученикам.

- Дашборд: список учеников, % завершения курса, последняя активность.

- Рассылка объявлений всему классу.

- Жёсткие границы: без оценок (это школа), без редактирования контента, без доступа к личной переписке ученика.

5.6 Логика репетитора (в PWA)

- Привязка к своим ученикам по коду — плоский список (без классов).

- Детализация по каждому ученику: прогресс, серия дней, ошибки.

- Жёсткие границы: без оценок, без редактирования контента, без доступа к чужим ученикам и к личной работе ученика.

- Доходимость в РФ онлайн-образовании — 20–30%. 60%+ — наше УТП.

- 15 мин/день — научный стандарт (Duolingo, A/B-тесты на 100+ млн пользователей).
