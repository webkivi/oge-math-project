# Протокол совместной работы Claude Code и Codex

Версия: v2
Дата: 2026-06-27
Проект: oge-math-project

## 0. Назначение

Два инструмента — одна кодовая база:

- **Claude Code** — архитектура, длинная реализация, мастер-документы.
- **Codex** — ревью, git-операции, Low/Mid-risk по принятой спеке.

Цель: экономить токены в Claude Code без потери архитектурной дисциплины.
Между инструментами передаётся НЕ история чата, а короткий формальный handoff-файл.

Этот документ не заменяет Methodology, Project Brain, CLAUDE.md/AGENTS.md и спеки.
Он регулирует только маршрутизацию задач.

## 3. Разделение зон

### Claude Code ведёт:
- А3 Архитектор, А4 Критик системы
- High-risk задачи
- Новое поведение без принятой спеки
- Изменения контрактов CSV / FSM / API / permissions / data model
- Обновления Brain / Methodology / CLAUDE.md
- Архитектурные развилки до начала кода
- Длинные многошаговые реализации

### Codex ведёт:
- Ревью кода и дифа (по рубрике А5 — читает `prompts/Агент_5_Ревьюер_кода_v1.txt`)
- Git-операции: push, fetch, merge, compare, remote-check
- Синхронизация GitHub / GitLab
- Узкие Mid-risk задачи по принятой спеке
- Low-risk исправления
- Независимая инженерная проверка перед возвратом в Claude Code

---

## 4. Триггеры передачи в Codex

Claude Code останавливается и пишет handoff, если выполнено хотя бы одно:

1. Диф готов — нужно ревью кода.
2. Следующий шаг — git-операция или синхронизация с GitLab.
3. Спека принята, оставшаяся работа узкая и локальная.
4. Нужно проверить реальное состояние файлов, веток или тестов.
5. Нужна независимая вторая проверка перед коммитом.

**Триггер 6 (лимиты) удалён:** упирание в лимиты само по себе не является триггером —
только если одновременно выполнен один из пунктов 1–5.

---

## 5. Когда передавать в Codex нельзя

Передача запрещена, если:
- поведение новое и спеки ещё нет
- меняются контракты данных
- проектируются права, ПД или приватность
- создаётся новая роль или новый FSM
- есть неразрешённая продуктовая развилка
- требуется обновление Brain / Methodology
- архитектурная неоднозначность не снята

## 7. Механика handoff через файлы (без копипаста)

Вместо копирования текста между окнами — handoff пишется в файл.

**Файлы-буферы (в .gitignore, не коммитятся):**
- `docs/handoff_out.md` — Claude Code пишет задачу для Codex
- `docs/handoff_in.md` — Codex пишет отчёт для Claude Code

**Цикл:**

```
1. Claude Code ведёт задачу до триггера.
2. Claude Code пишет handoff в docs/handoff_out.md и говорит:
   STOP: handoff записан в docs/handoff_out.md
3. Ты переключаешься в Codex и говоришь:
   Прочитай docs/handoff_out.md и выполни задачу.
4. Codex выполняет работу и пишет отчёт в docs/handoff_in.md.
5. Ты возвращаешься в Claude Code и говоришь:
   Прочитай docs/handoff_in.md и продолжай.
6. Claude Code читает отчёт и продолжает следующую стадию.
```

Ты переключаешь окна — не копируешь текст.

---

## 8. Ревью в Codex: как работает рубрика А5

Codex не запускает промпт А5 автоматически.
Чтобы ревью шло по рубрике А5, Claude Code прописывает в handoff_out.md:

```
Перед ревью прочитай prompts/Агент_5_Ревьюер_кода_v1.txt и следуй его рубрике.
```

Это единственная строка, которая даёт Codex правила ревью.
Без неё Codex делает инженерное ревью по собственному суждению — не по стандарту А5.

---

## 9. Критерии хорошего handoff

**Хороший:**
- одна задача
- чёткая граница (что можно, что нельзя)
- ссылки на файлы, не копирование их содержимого
- понятный критерий готовности

**Плохой:**
- «посмотри весь проект»
- «доделай по смыслу»
- пересказ всей истории сессии
- задача без критерия готовности

---

## 10. Признаки что протокол работает

- Claude Code тратит меньше токенов на ревью и синхронизацию
- Codex не придумывает архитектуру и не меняет контракты
- handoff_out.md помещается в 20–30 строк
- Принятые спеки остаются главным управляющим артефактом
- Ты переключаешься между инструментами без повторного объяснения проекта
